We are ready to build **Email Intelligence Layer v1** for Value Truck.

This is NOT a generic inbox feature.
This is a structured pipeline that:
1) normalizes all app-side email we already send/receive,
2) extracts intents and signals from those messages (customer + carrier),
3) writes those signals into the right places: account score / NBA, carrier hub / carrier NBAs, lane work queue, and win-tracking.

We will NOT (yet) integrate with every external inbox (e.g., Outlook/Gmail for every rep).
In v1, we focus on email that already flows through the Value Truck app:
- carrier procurement emails
- future customer emails sent/received via the platform
- any existing send-from-platform lanes we already log

--------------------------------------------------
1. GOAL & SCOPE
--------------------------------------------------

Goal:
Create a reusable **Email Intelligence Layer** with three pieces:
- a normalized email message store
- an intent/signal extraction pipeline
- small, safe integrations into:
  - Account score / NBA inputs (customer side)
  - Carrier Hub / carrier NBAs (carrier side)
  - Lane Work Queue / tasks where appropriate

Scope:
- v1 is **app-side** only (emails passing through the platform).
- Do NOT attempt full Outlook/Gmail sync in this task.
- Do NOT redesign the NBA engine or account scoring, but make it easy to consume email signals.

--------------------------------------------------
2. UNIFIED EMAIL MESSAGE STORE
--------------------------------------------------

First, normalize email messages.

If an `email_messages`-like table already exists, extend/reuse it.
If not, add a new table (name suggestions: `email_messages` or `messages_email`).

Minimum fields:
- id (PK)
- threadId (group messages into conversations)
- direction: 'inbound' | 'outbound'
- sentAt / receivedAt timestamps
- fromAddress
- toAddresses (array/json)
- ccAddresses (array/json, optional)
- subject
- bodyText (plain text)
- bodyHtml (optional)
- rawHeaders (optional)
- mailbox / channel / source (if needed)
- orgId (if multi-org is relevant)
- userId (sending/owning user, if outbound)
- linkedAccountId (nullable)
- linkedCarrierId (nullable)
- linkedLaneId (nullable)
- linkedLoadId (nullable)
- linkedTaskId / linkedNbaId (nullable)

Design principles:
- This table is the source of truth for message content.
- Keep the schema simple but expressive enough for future use.

Tasks:
- Add schema + migration (via runMigrations.ts and any associated SQL/journal).
- Add storage methods to IStorage / DatabaseStorage:
  - insertEmailMessage
  - updateEmailMessageLinks (to attach account/carrier/lane/load)
  - getEmailMessagesForAccount / Carrier / Lane / Task as needed

For v1, it’s OK if linkage is done by existing flows that already know the account/carrier/lane.

3. EMAIL INTENT & SIGNAL TAXONOMY
--------------------------------------------------

Define a small, explicit taxonomy of **intents** and **signals** we care about.

Customer-side email intents (examples):
- meaningful_touchpoint
- pricing_request
- objection (with subtype: budget, service, timing, competitor, internal_priority)
- service_complaint
- positive_feedback
- new_opportunity (new lane, new location, new product, etc.)
- stalled_thread (detected by time + content)
- urgency_signal (deadline, “ASAP”, risk language)
- closed_won_indicator
- closed_lost_indicator

Carrier-side email intents (examples):
- lane_offer (we can cover X→Y)
- lane_decline
- capacity_available (region/equipment/date)
- capacity_unavailable
- new_lane_preference
- new_equipment / new_region
- price_pushback
- service_issue
- paperwork/compliance step
- soft_commitment / hard_commitment

Represent these as:
- string enums / union types in code,
- optional reference tables in DB if needed.

We do NOT need a huge taxonomy in v1; it just needs to be:
- explicit,
- documented,
- easy to extend.

4. EMAIL SIGNAL STORAGE
--------------------------------------------------

Create a table to store extracted signals per message, e.g. `email_signals`.

Suggested fields:
- id
- emailMessageId (FK -> email_messages)
- orgId (if relevant)
- direction (inbound/outbound)
- intent (string or enum)
- intentSubtype (optional, e.g., objection_budget)
- actorType: 'customer' | 'carrier' | 'internal' | 'unknown'
- primaryEntityType: 'account' | 'carrier' | 'lane' | 'load' | 'task' | 'unknown'
- primaryEntityId
- confidenceScore (0–1 or 0–100)
- extractedData JSONB (structured payload: e.g., lane description, dates, locations, amounts)
- createdAt

Tasks:
- Add schema + migration for `email_signals`.
- Add IStorage / DatabaseStorage methods:
  - insertEmailSignals(emailMessageId, signals[])
  - getEmailSignalsForAccount / Carrier / Lane / Load (basic queries)

--------------------------------------------------
5. INTENT & ENTITY EXTRACTION PIPELINE
--------------------------------------------------

Implement a service (e.g., `emailIntelligenceService.ts`) that:

- Takes an email message (subject + bodyText + metadata).
- Detects:
  - actorType,
  - intent(s),
  - optional intentSubtype(s),
  - extracted entities:
    - for customer: lanes, locations, dates, volumes, price mentions, competitor names, sentiments
    - for carrier: origin/destination, equipment, date ranges, willingness (yes/no/maybe), rate hints
- Returns one or more `EmailSignal` objects ready to store.

Implementation notes:
- Use existing OpenAI client / LLM integration patterns if present.
- Keep the extraction deterministic in terms of output schema (no freeform text fields beyond explanation).
- Include rules to:
  - ignore signatures / quoted history where possible,
  - avoid double-counting the same intent on a thread unless clearly new.

For v1, a simple call per email is fine. Later we can batch or stream.

Tests:
- Provide fixtures of synthetic email bodies and test that:
  - pricing requests are detected correctly,
  - objections are classified with the right subtype,
  - service complaints vs. neutral status updates are distinguished,
  - carrier lane offers produce structured lane/region/equipment data in extractedData.

--------------------------------------------------
9. GUARDRAILS & SAFETY
--------------------------------------------------

Important:
- Do NOT auto-overwrite account, contact, or carrier core fields directly from email in v1.
- Instead:
  - use email signals as:
    - inputs to scores and NBAs,
    - or as suggestions (staging tables) for profile updates.
- Avoid spam:
  - do not create dozens of NBAs or tasks from one noisy thread.
  - dedupe signals per thread where appropriate.

Implement:
- Simple dedupe rules:
  - e.g., only one `pricing_request` signal per thread per day,
  - only one `service_complaint` signal per thread unless clearly separate issues.
