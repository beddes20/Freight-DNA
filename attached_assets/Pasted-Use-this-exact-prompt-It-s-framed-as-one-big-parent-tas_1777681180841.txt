Use this exact prompt. It’s framed as one big parent task with clear sub-tasks, guardrails, and rollout discipline.

> Build a new Freight-DNA feature called **Context Notes** (working name) — a structured, in-platform collaboration layer that lets reps and managers tag teammates directly on the exact object they’re looking at, leave a note, and bring the receiver back to that exact place in the app.
>
> This is **not** a generic notes feature and **not** a sticky-note system.  
> It should be a **contextual collaboration system** attached to real workflow objects, with mentions, replies, acknowledgment, resolution, deep links, and optional conversion to task.
>
> Treat this as **one parent task with multiple sub-tasks** and work through it end-to-end until the full v1 is complete, tested, and documented.
>
> ---
>
> ## Product intent
>
> The goal is to reduce friction in coordination.
>
> Today, if someone wants input or action from a teammate, the alternatives are weak:
> - create a detached task,
> - send a Slack/email,
> - or rely on memory.
>
> I want a user to be able to tag someone **in context** on the exact quote, conversation, lane, load, customer, or carrier record they’re looking at, add a short note, and have the receiver:
> - get notified,
> - open the exact linked location,
> - reply, acknowledge, resolve, or convert it into a task.
>
> This feature should make the platform feel more operational, collaborative, and accountable — without turning into noise.
>
> ---
>
> ## Core product rules
>
> 1. **Anchored, not free-form**
> - Notes must be attached to a stable anchor/object.
> - Do **not** implement “click literally anywhere on the page and drop a pin” in v1.
> - v1 should only support structured anchors tied to real entities.
>
> 2. **Collaboration, not clutter**
> - This is for teammate coordination.
> - It should support:
>   - mention,
>   - reply,
>   - acknowledge,
>   - resolve,
>   - convert to task.
> - It should not become a second messy inbox.
>
> 3. **Deep-link back to exact context**
> - When someone receives a note, clicking it should take them back to:
>   - the correct page,
>   - the correct object,
>   - and ideally the correct panel/section/row if technically feasible.
>
> 4. **Permissions matter**
> - Users should only see notes for records they are allowed to view.
> - Mentions should not leak visibility into hidden/private records.
>
> 5. **Tasks still matter**
> - Notes do not replace tasks.
> - If follow-through is required, there should be a clean path to convert a note into a real task.
>
> ---
>
> ## v1 scope — supported anchors
>
> For v1, support Context Notes on these object types only:
>
> - Quote Requests / quote opportunities
> - Conversation threads
> - Available Freight rows
> - Lane Work Queue rows
> - Customer record pages
> - Carrier record pages
> - Load detail pages (if a stable route/object exists)
>
> Out of scope for v1:
> - arbitrary click-anywhere page annotations,
> - charts/KPIs with pixel-position anchors,
> - canvas-like comments,
> - document/PDF annotations,
> - comments on transient UI elements that do not have stable IDs.
>
> ---
>
> ## Required user actions
>
> For each note:
>
> - Create a note on a supported object.
> - Mention one or more teammates via `@name`.
> - Add plain-text note content.
> - View threaded replies.
> - Acknowledge.
> - Resolve.
> - Re-open if needed.
> - Convert to task.
>
> Each note should also have:
> - author,
> - created time,
> - updated time,
> - anchor type,
> - anchor id,
> - deep link / route payload,
> - mentioned user ids,
> - status (`open`, `acknowledged`, `resolved`),
> - optional action type (`FYI`, `Question`, `Please review`, `Please handle`, `Decision needed`).
>
> ---
>
> ## UX expectations
>
> I do **not** want this sprayed all over the UI.  
> The experience should be clean and reusable.
>
> Design expectations:
>
> 1. **Entry point**
> - Each supported object/row/detail page should have a small, consistent “Comment / Note / Mention” entry point.
> - In dense tables, this may live in a row action menu or small icon.
> - On detail pages, it can live in an activity rail or contextual side panel.
>
> 2. **Composer**
> - Lightweight composer:
>   - text input,
>   - mention autocomplete,
>   - action-type dropdown,
>   - submit.
> - Fast and minimal, not a huge modal.
>
> 3. **Display**
> - Notes should appear in a compact thread view tied to the anchored object.
> - Show status clearly.
> - Show mentions clearly.
> - Replies should stay nested/threaded.
>
> 4. **Notification destination**
> - Add a lightweight “Mentions / Context Notes” inbox or notification center entry so users can see:
>   - open items,
>   - acknowledged,
>   - resolved,
>   - assigned/mentioned to me.
>
> 5. **Deep link behavior**
> - Clicking a notification should land the user on the correct route and reveal the relevant object/thread/note.
> - If the object still exists but is filtered out by default, the app should attempt to reveal it safely.
>
> 6. **Convert to task**
> - One-click “convert to task” from a note.
> - Preserve backlink from task → original anchored object.
>
> ---
>
> ## Technical architecture
>
> Please design this as a reusable platform primitive, not six custom implementations.
>
> I want:
>
> ### A. Shared data model
> Create a reusable Context Notes model, likely something like:
> - `context_notes`
> - `context_note_mentions`
> - `context_note_replies` (or a unified threaded model)
> - `context_note_events` (optional, if helpful for audit trail/status changes)
>
> The model should support:
> - anchor type + anchor id,
> - route/deep-link payload,
> - author,
> - mentioned users,
> - status,
> - action type,
> - task conversion link if applicable,
> - auditability.
>
> ### B. Shared service layer
> Build shared server helpers/services for:
> - create note,
> - reply,
> - acknowledge,
> - resolve,
> - reopen,
> - list notes by anchor,
> - list notes for “mentioned to me,”
> - convert note to task,
> - notification fanout.
>
> ### C. Shared client primitives
> Build reusable client primitives/components/hooks for:
> - `ContextNoteComposer`
> - `ContextNoteThread`
> - `ContextNoteBadge`
> - `useContextNotes(anchor)`
> - `MentionsInbox` or equivalent
>
> ### D. Deep-link contract
> Create a stable route payload/deep-link contract by anchor type so this does not become brittle.
> I want a clear map for each supported object type:
> - route,
> - lookup parameters,
> - how the UI reveals/highlights the anchored object,
> - fallback behavior if exact reveal is not possible.
>

> ## Important product constraints
>
> - Do **not** overbuild v1.
> - Do **not** make this a giant social feed.
> - Do **not** create floating sticky notes pinned to arbitrary x/y coordinates.
> - Do **not** create six different note systems for different tabs.
> - Do **not** hide critical workflow behind comments.
>
> Instead:
> - make it structured,
> - make it anchored,
> - make it reusable,
> - make it accountable,
> - make it easy to escalate into a task when needed.
>

>
> ## What good looks like
>
> A rep is on a Quote Request and tags a manager:
> - “@Ben customer says incumbent is at 1925 and needs answer by 4pm — should we stay firm or push?”
>
> Ben gets a notification, clicks it, lands on that exact quote, sees the full context, replies inline, and the rep can mark it resolved or convert it to a task if more work is needed.
>
> Another example:
> - Someone is on a lane in LWQ and tags procurement:
>   - “@Taylor this lane keeps rejecting at current target — can you review carrier mix?”
>
> Taylor clicks directly into that lane and works from the anchored context, not from a disconnected task title.
>
> That is the experience I want.
