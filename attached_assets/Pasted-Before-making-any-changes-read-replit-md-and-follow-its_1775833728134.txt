Before we implement any larger Carrier Hub email-enrichment feature, I want to harden and document the CURRENT two-way email process first.

We recently got portal email working in both directions (send + receive). That is good progress, but I do NOT want to build a large AI-driven carrier profile enrichment layer on top of a process that is only partially solid.

This task is NOT to build the enrichment feature yet.
This task is to inspect, validate, and tighten the current email workflow so we know the foundation is reliable.

Goal:
Audit and harden the current two-way email process end-to-end so we can confidently build future features on top of it.

What I want from this task:
1. trace the current send/receive email flow in code
2. identify the exact source-of-truth tables/models used for outbound + inbound messages
3. confirm how threading/conversation association currently works
4. confirm how emails are associated to carriers and/or lanes
5. identify gaps, risks, and edge cases
6. make targeted fixes if the current implementation is incomplete or fragile
7. add/update tests for the current foundation
8. return a concise architecture summary and readiness assessment

Please inspect the existing implementation carefully before changing code.

Questions I need answered:
- What exact files/functions handle outbound email send?
- What exact files/functions handle inbound email receive/webhook processing?
- Where are outbound emails stored?
- Where are inbound emails stored?
- Is there one conversation/thread model or multiple partial logging models?
- How is an inbound reply matched back to:
  a) a prior outbound message,
  b) a carrier,
  c) a lane, if applicable?
- What identifiers are currently persisted (messageId, threadId, provider ids, recipients, sentAt, receivedAt, delivery state, etc.)?
- Is raw email content stored? cleaned content? both?
- What happens if the sender email does not exactly match the email we originally sent to?
- What happens if multiple contacts exist for the same carrier?
- What happens if the inbound email cannot be confidently matched to a carrier or thread?
- What happens with duplicate inbound webhook events?
- What happens with bounces, failures, or malformed payloads?

Hardening requirements:
Please make the current email process solid and explicit in these areas if it is not already:

A. Message persistence
- Ensure outbound and inbound messages are persistently stored in a clear, queryable way.
- Preserve enough metadata for auditing and future AI processing:
  - provider message id(s)
  - thread/conversation id if available
  - from
  - to/cc/bcc if available
  - subject
  - raw body or raw payload reference
  - cleaned/plaintext body used for display/parsing
  - sent/received timestamps
  - delivery status / receive status
  - related carrier id if known
  - related lane id if known
  - direction: outbound | inbound

B. Threading / conversation association
- Make sure replies can be grouped into a coherent conversation.
- If the app currently has partial logging in multiple places, consolidate or clearly link them.
- If lane-linked outreach logs already exist, do not break them — but make sure the broader email conversation model is still usable for future carrier hub intelligence.

C. Carrier association
- Define and centralize the logic for matching inbound sender -> carrier/contact.
- Reuse existing contact/carrier data where possible.
- Be explicit about confidence / fallback behavior when sender matching is ambiguous.

D. Lane association
- If an inbound reply is part of a lane outreach thread, preserve that lane link.
- If an inbound email exists outside a lane-specific context, do not force a fake lane link.

E. Failure and ambiguity handling
- Duplicate webhook events should not create duplicate messages.
- Unknown senders should be stored in a recoverable/unmatched state, not dropped.
- Malformed payloads should fail safely and be logged.
- Bounces / failed sends should be visible and auditable.

F. UI visibility
- If there is already a History or communication view, make sure the current two-way flow is actually visible there in a usable way.
- I do not need a major new inbox UI in this task, but I do need confidence that a user can inspect what was sent and what was received.

G. Tests
Add or update tests for at least:
- outbound email logging
- inbound email logging
- reply matched back to existing conversation/thread
- reply matched to the correct carrier
- unmatched inbound sender is stored safely
- duplicate inbound event does not duplicate records
- lane-linked email retains lane association
- non-lane email does not get fake lane linkage
- failures/bad payloads are handled safely

Deliverables:
Return:
1. short summary of the current architecture
2. exact files inspected
3. exact files changed
4. data models/tables currently used for email
5. how threading currently works
6. how carrier matching currently works
7. what gaps were found
8. what fixes were made
9. tests added/updated
10. a readiness assessment for the future Carrier Hub enrichment feature:
   - ready now
   - ready after small follow-up
   - not ready yet
