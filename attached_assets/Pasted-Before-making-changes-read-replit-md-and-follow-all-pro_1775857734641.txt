2. BACKEND – CARRIER INTELLIGENCE SUMMARY API
--------------------------------------------------

Add an internal route, e.g.:

GET /api/internal/carriers/:carrierId/intelligence

Return:

{
  carrier: {
    id,
    name,
    mcNumber,
    dotNumber,
    primaryEquipment,
    homeRegions,
    ... (basic profile fields)
  },
  history: {
    topLanes: [ { origin, destination, loadsCount }, ... ],
    topRegions: [ { region, loadsCount }, ... ],
    equipmentMix: [ { equipmentType, loadsCount }, ... ]
  },
  declaredPreferences: {
    lanes: [...],
    regions: [...],
    equipment: [...]
  },
  marketSignals: [
    { id, type, scope, severity, createdAt, description }
  ],
  suggestions: {
    pending: [...],
    accepted: [...],
    rejected: [...]
  }
}

Use existing history and market-signal tables to populate:
- top lanes/regions/equipment (from TMS / lane history).
- current Market NBAs / signals for this carrier.

Use carrier_intel_suggestions to populate:
- pending/accepted/rejected.

Apply the same auth & multi-org guardrails as other internal carrier routes.

--------------------------------------------------
3. BACKEND – ACCEPT/REJECT ENDPOINTS
--------------------------------------------------

Add endpoints:

- POST /api/internal/carriers/:carrierId/intelligence/suggestions/:suggestionId/accept
- POST /api/internal/carriers/:carrierId/intelligence/suggestions/:suggestionId/reject

Behavior:
- validate carrierId ownership/org
- update `status`, `acceptedAt`/`rejectedAt`, `acceptedByUserId`/`rejectedByUserId`
- optional comment in request body
- return updated suggestion

Do NOT mutate core carrier profile in this task.
We are just managing the suggestion lifecycle.

--------------------------------------------------
4. FRONTEND – CARRIER HUB INTELLIGENCE TAB
--------------------------------------------------

In the Carrier Hub UI:

A) Add an “Intelligence” tab or section in the carrier detail view.

When selected:
- call the intelligence API for that carrier.
- show a loading/skeleton state while fetching.

B) Layout (desktop-first, responsive):

Sections:
1) “Lane & Region History”
   - show top lanes with counts (chip/table)
   - top regions
   - equipment mix

2) “Declared Preferences”
   - lanes / regions / equipment the system already considers as declared/canonical.

3) “Suggested Intelligence” (Pending)
   - list of pending suggestions grouped by suggestionType:
     - lane_preference
     - capacity_available/unavailable
     - equipment_capability
     - region_preference
     - price_sensitivity
     - service_risk
   - each item shows:
     - a concise description
       - e.g. “AZ → PNW, dry van, weekly”
     - source:
       - from email: date + “View email” link (if we have a thread URL or at least a threadId)
       - from market signal: type + region
     - confidenceScore (visual hint; not necessarily numeric)
     - createdAt relative time (“3 days ago”)
     - Accept / Reject buttons (and optional comment field or modal)

4) “History & Decisions”
   - collapsed section listing accepted & rejected suggestions:
     - what was accepted/rejected
     - by whom
     - when

C) UX rules:
- Accept/Reject calls hit the new endpoints and update local state without full reload.
- Provide minimal but clear labels; avoid overwhelming the user.
