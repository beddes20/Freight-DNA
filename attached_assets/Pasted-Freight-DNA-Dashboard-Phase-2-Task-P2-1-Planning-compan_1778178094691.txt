Design the migration / rollout plan

Outline the phased steps, for example:

Step 1 — Add new table:

Create company_financial_aliases table.

Backfill from existing companies.financialAlias where possible:

For each non-null alias, insert a row into company_financial_aliases with source='legacy_column'.

Step 2 — Dual-read:

New resolver function reads from company_financial_aliases first; still falls back to companies.financialAlias for legacy safety.

No behavior change yet for callers.

Step 3 — Dual-write:

Any place that writes/updates aliases writes to both:

the new table,

and the legacy companies.financialAlias, to keep them in sync.

Step 4 — Shadow mode:

New resolver runs in parallel (flagged per org) and logs mismatches between the old fuzzy match and the new strict match.

Step 5 — Cutover:

For opted-in orgs, use the new resolver only.

Keep the old column as a backup for a defined period.

Step 6 — Deprecate:

Once confidence is high, stop writing to companies.financialAlias and eventually drop or freeze it.
