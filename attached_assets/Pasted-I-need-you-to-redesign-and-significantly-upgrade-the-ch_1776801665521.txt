I need you to redesign and significantly upgrade the chatbot / AI assistant inside my existing Freight-DNA platform.

Current problem:
The chatbot is not intuitive enough, not useful enough, and not deeply connected enough to the data in the platform. Right now it is nowhere near where we want it to be.

Main goal:
Turn the current chatbot into a highly intuitive in-platform AI copilot that can analyze any relevant data in Freight-DNA, answer questions in plain language, guide users to the right workflow, and safely perform or recommend actions when appropriate.

This is NOT just a generic chatbot refresh.
This should become a trusted operational copilot for a freight brokerage / logistics workflow.

Important context:
- Freight-DNA is an existing platform, not a greenfield app.
- The platform contains operational, customer, carrier, shipment, lane, outreach, email, task, and activity data.
- Some data is structured, and some is unstructured or semi-structured.
- The chatbot should be able to use AI where appropriate, but the solution should not depend on AI for things that can be solved more reliably with deterministic system logic.
- I care about trust, speed, accuracy, explainability, and operational usefulness.
- I do not want a black-box assistant that gives vague answers with no grounding.
- I want this to feel native to the platform and actually help reps, ops, managers, and leadership get answers faster and take better actions.

Your first task:
Do NOT start coding immediately.

First inspect the current codebase and return:
1. findings
2. current chatbot architecture
3. what data sources the bot can and cannot currently access
4. current limitations / failure points
5. recommended target architecture
6. exact files / services / routes / components to change
7. risks / assumptions
8. phased implementation plan
9. validation plan

Then pause or checkpoint before major implementation if risk is high.

What the new copilot should be able to do:

A) Understand natural language intuitively
The assistant should understand real user questions, not force users to speak in robotic keyword phrases.

Examples:
- What freight do we have available in Texas next week?
- Which carriers are best for Chicago to Atlanta dry van?
- Why has ACH Foods had no growth lately?
- Show me customers with declining touchpoints but rising shipment volume
- What lanes are most consistent for this customer?
- Which carriers have replied but have not been followed up with?
- What happened on this account in the last 30 days?
- Who owns this freight?
- What should I do next on this account?
- Summarize all activity for this carrier
- What emails mention rejected tenders this week?
- Which customers are growing but still underpenetrated?
- Find recurring freight for this customer
- Which reps have open follow-ups on strategic accounts?

The bot should be able to handle:
- direct questions
- follow-up questions
- vague questions
- comparative questions
- summary requests
- trend / pattern questions
- workflow guidance questions
- “what should I do next?” questions

B) Access and analyze all relevant platform data
The bot should be able to analyze data across the platform, including where appropriate:
- shipments
- open / available freight
- booked freight
- customers
- carriers
- lanes
- recurring freight patterns
- emails / outreach
- tasks / reminders / follow-ups
- touchpoints / call logs
- account growth signals
- ownership / assignments
- scorecards / weighted metrics
- notes / activity history
- user / rep activity
- any existing dashboards or analytics sources

If there are data silos, duplicated logic, missing joins, or inaccessible datasets, identify them and fix them.

Build a clear data-access strategy that separates:
- structured queryable data
- semi-structured activity data
- unstructured text data like emails, notes, and messages

Use deterministic queries where possible.
Use AI / LLM reasoning where useful for summarization, interpretation, classification, entity extraction, and cross-source synthesis.

C) Provide grounded, trustworthy answers
I want answers to be useful and trustworthy.

The assistant should:
- explain where an answer came from
- identify which records, entities, or date range it used
- show confidence / uncertainty where needed
- avoid hallucinating
- avoid answering from incomplete data without saying so
- distinguish between hard facts, inferred insights, and recommendations
- clearly say when the data is insufficient or ambiguous

If helpful, answers should include:
- linked records
- relevant filters
- supporting metrics
- recent activity summaries
- suggested next actions

D) Be action-oriented, not just informational
The copilot should not only answer questions.
It should also help users get work done.

Examples:
- draft an email based on account context
- suggest the top next actions on an account
- create a follow-up task
- open the relevant lane, customer, carrier, or shipment record
- prefill filters in the proper queue or dashboard
- summarize a thread before a call
- identify stale opportunities
- recommend carriers for a load or lane
- highlight risks or missing follow-up

If direct system actions are too risky for v1, then first implement “guided actions”:
- recommendations
- deep links
- drafts
- prefilled workflows
- confirm-before-execute actions

E) Role-aware behavior
The assistant should respond differently based on who is asking.

Examples:
- reps need actionable account and lane guidance
- ops needs freight, coverage, and execution help
- managers need team visibility and coaching insight
- leadership needs summaries, trends, exceptions, and performance views

If role/permission logic already exists, reuse it.
The bot must respect user permissions and organizational visibility boundaries.
Never expose data a user should not see.

F) Conversational memory and context
The assistant should remember context within the conversation:
- if the user says “show me the top customers in Texas”
- then asks “which ones are declining?”
- then says “summarize the first one”
the assistant should preserve context naturally.

It should also be able to resolve references like:
- this customer
- that carrier
- those loads
- the first one
- last week
- next month
- the rep I mentioned earlier

If full persistent memory is too risky, start with strong session memory and contextual follow-up handling.

G) Strong query orchestration
Build a smart orchestration layer so the assistant decides how to answer:
- direct SQL / ORM / service queries for structured facts
- search / retrieval for notes, emails, and text-heavy content
- hybrid retrieval + AI reasoning for cross-source questions
- deterministic business logic for workflows, rankings, and calculations
- LLM summarization only after the right evidence has been gathered

Do not build a chatbot that sends everything to the LLM first.
The assistant should choose the right tool path based on the question.

H) Suggested intents / capabilities
The assistant should be able to support at least these categories:
- search for records
- summarize account or carrier history
- analyze freight patterns
- explain score / ranking logic
- identify opportunities
- identify risks / exceptions
- surface follow-up gaps
- compare customers / carriers / lanes
- recommend next best actions
- draft emails or call prep
- create tasks or reminders
- answer metric / trend questions
- help users navigate the platform

If these should be implemented as skills, tools, handlers, or domain modules, structure it cleanly.

I) UX / product experience
Make the chatbot feel intuitive and embedded in the platform, not like a toy widget.

Desired UX ideas:
- assistant panel or expandable workspace
- suggested prompts based on page context
- contextual questions depending on where the user is
- examples when empty
- follow-up chips or suggested next questions
- answer cards with links into relevant records
- tables, bullets, or mini summaries where useful
- loading states that feel intelligent, not fake
- graceful fallback when the assistant cannot answer fully
- visible source grounding when useful
- page-aware behavior so if I am on a customer page, the bot knows that context

If the current UI is weak, redesign it to be operationally strong and intuitive.
Keep it clean and native to Freight-DNA.

J) Safety / guardrails
Add guardrails so this becomes a trusted tool.

Include:
- permission-aware data access
- auditability of actions
- confirmation before sensitive actions
- safe fallback when confidence is low
- protection against broad unsupported claims
- clear error states
- prevention of cross-account / cross-user data leakage
- logging of failures, unresolved intents, and poor-answer cases
- instrumentation so we can improve the bot over time

K) Analytics / feedback loop
Add instrumentation so we can improve the copilot after launch.

Track:
- top user questions
- unanswered questions
- low-confidence answers
- time to answer
- most used actions
- follow-up usage
- abandonment
- thumbs up / down if appropriate
- which data sources are most useful
- where the bot fails due to missing data vs poor reasoning vs UI confusion

L) Technical expectations
Work from the current app and current architecture.
Do not give me a generic greenfield chatbot architecture unless necessary.
Inspect existing:
- frontend chatbot components
- backend chat routes
- AI service layers
- data access layers
- analytics
- permissions
- search or indexing
- email / note / activity models
- page context sources

Then propose the best practical architecture for this app.

Possible architecture direction if needed:
- intent/router layer
- structured query layer
- retrieval/search layer for text-heavy data
- AI summarization / reasoning layer
- action layer
- response formatting layer
- observability layer

But only implement what fits the current system cleanly.

M) Performance expectations
The assistant needs to feel fast enough to use in real workflows.

Optimize for:
- fast response on simple factual queries
- progressive responses for harder queries
- caching where appropriate
- minimal unnecessary LLM calls
- smart retrieval scope
- good handling of large data volumes
