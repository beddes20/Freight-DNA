I need you to perform a full-system audit of this existing platform and do not treat this as a light review or surface-level pass.
A large number of features, updates, and changes were added recently, and I need you to assume there are likely hidden issues, weak integrations, regressions, incomplete merges, inefficient logic, and disconnected user experiences unless you prove otherwise.
Your job is to:
•	identify issues,
•	uncover hidden issues,
•	fix the issues you find,
•	verify the fixes actually work,
•	and then provide recommendations for both the fixes made and the overall platform improvement opportunities.
This is not just a bug check. This is a deep platform hardening pass.
I want you to approach this like:
•	a senior QA engineer,
•	a senior full-stack architect,
•	a product-minded systems thinker,
•	and an aggressive end user trying to break the app.
The core objective is to make this platform feel fully connected, intuitive, stable, and polished.
I do not want separate features that happen to exist in the same platform.
I want a unified system where modules, workflows, actions, records, automations, dashboards, messaging, and supporting tools all behave like they belong to one product.
Important mindset:
If something appears to work, do not assume it is fully correct.
Validate whether it:
•	saves correctly,
•	updates correctly,
•	propagates correctly,
•	displays correctly elsewhere,
•	handles errors correctly,
•	and behaves consistently with the rest of the platform.
Execution requirements:
1.	Start by auditing the existing system before making broad code changes.
2.	Map the major modules, routes, workflows, data objects, integrations, state dependencies, automations, and feature relationships.
3.	Identify where recent additions may have introduced:
•	bugs,
•	regressions,
•	partial implementations,
•	duplicate logic,
•	stale code,
•	broken integrations,
•	inconsistent naming,
•	inconsistent UI behavior,
•	bad validation,
•	poor error handling,
•	or disconnected workflows.
Then proceed in phases.
PHASE 1 — PLATFORM DISCOVERY
Inspect the full codebase and produce a working understanding of:
•	core architecture
•	frontend structure
•	backend/service structure
•	major feature areas
•	routes/pages/views
•	shared components
•	shared state or data dependencies
•	database/storage interactions
•	automations, triggers, notifications, and background logic
•	recently changed or newly introduced features if visible
Create a risk map showing:
•	the most fragile areas,
•	the most tightly coupled areas,
•	the most likely regression points,
•	and the areas most likely to appear functional while still being broken underneath.
PHASE 2 — ISSUE HUNT
Systematically search for:
•	bugs
•	regressions
•	incomplete merges
•	broken flows
•	inconsistent field mappings
•	bad assumptions in the UI
•	stale or duplicated components
•	orphaned logic
•	silent failures
•	missing feedback states
•	validation gaps
•	data handoff failures
•	permission issues
•	performance inefficiencies
•	confusing or fragmented UX
Be suspicious of:
•	forms that submit but do not persist correctly
•	records that save but do not update related views
•	dashboards that lag behind source data
•	filters/search/sorts that appear correct but are not
•	actions that update one module but not another
•	old logic still controlling new workflows
•	multiple entry points that should do the same thing but behave differently
•	UI that gives the impression of completion without real completion
PHASE 3 — FULL INTERACTION TESTING
I want an aggressive testing pass across the platform.
Test every reasonable interaction, including:
•	every button
•	every checkbox
•	every toggle
•	every dropdown
•	every tab
•	every modal
•	every link
•	every save/cancel/back path
•	every create/edit/delete/archive path
•	every filter/search/sort path
•	every bulk action
•	every import/export/upload/download path
•	every notification or message flow
•	every automation trigger
•	every assistant/chat/tool workflow if present
•	every role-based or restricted path if present
Also test:
•	empty states
•	loading states
•	failure states
•	edge-case inputs
•	duplicate inputs
•	null/blank inputs
•	malformed inputs
•	oversized inputs
•	refresh persistence
•	navigation transitions
•	cross-module updates
•	downstream effects after changes are made
If data can enter the system, trace where it should go.
If one feature should affect another, verify that it actually does.
PHASE 4 — FIX ISSUES
Fix the issues you find in a smart order:
1.	critical failures
2.	broken workflows
3.	data integrity and sync issues
4.	regressions
5.	poor validation and poor error handling
6.	UX confusion and fragmented behavior
7.	inefficiencies, duplication, and cleanup
Do not perform unnecessary large rewrites.
Prefer targeted, high-confidence fixes that improve stability, cohesion, and maintainability.
Where appropriate:
•	consolidate duplicate logic
•	improve consistency in naming
•	improve state handling
•	improve validation
•	improve system feedback
•	reduce disconnected behavior across modules
•	tighten how related features communicate
PHASE 5 — VERIFY EVERY FIX
Do not assume a fix is complete because the code compiles or the screen renders.
For every fix:
•	retest the original issue
•	verify the expected behavior now works
•	run regression checks on related areas
•	confirm upstream and downstream effects are correct
•	confirm persistence is correct
•	confirm displayed state matches actual stored state
•	confirm no new breakage was introduced
PHASE 6 — IMPROVEMENT RECOMMENDATIONS
After fixes are complete, provide recommendations in two categories:
A. Fix-specific recommendations
For each major issue fixed, provide:
•	what was fixed
•	why it likely happened
•	what safeguard should be added to prevent it from happening again
•	whether there are adjacent areas that should still be watched
B. Overall platform improvement recommendations
Provide broader suggestions for:
•	architecture
•	UX consistency
•	cross-feature cohesion
•	state/data flow reliability
•	automation reliability
•	test coverage
•	performance
•	maintainability
•	simplification of redundant flows
•	making the platform feel more unified and intuitive
OUTPUT FORMAT
Provide your output in this structure:
1.	System Map
•	main modules
•	key workflows
•	connected systems
•	likely recent additions
•	highest-risk areas
1.	Issues Identified
For each issue:
•	title
•	severity
•	category
•	where found
•	reproduction steps
•	expected result
•	actual result

1.	Fixes Completed
For each fix:

•	what changed
•	why it changed

1.	Fix Verification
•	original issue retested
•	related regression checks performed
•	results of revalidation
1.	Fix-Specific Suggestions
•	prevention ideas
•	future watchouts
•	follow-up improvements
1.	Overall Platform Improvements
•	structural improvements
•	UX improvements
•	integration improvements
•	performance improvements
•	maintainability improvements
•	recommended next hardening steps
NON-NEGOTIABLE RULES
•	Do not stop at obvious issues.
•	Investigate deeply enough to uncover hidden and secondary issues.
•	Do not mark work complete until issues are found, fixed, and verified.
•	If you fix something, explain the fix and suggest how to prevent recurrence.
•	If you notice broader product or architecture improvements, include them.
•	Focus on making the platform stable, intuitive, connected, and production-ready.
•	Assume incomplete work may be hiding behind apparently functional UI.
•	Be thorough, skeptical, and systematic.
Before making broad changes, first provide the system map, risk map, and audit plan.
Then execute the issue hunt.
Then fix issues.
Then verify the fixes.
Then provide suggestions for both the fixed items and overall platform improvement.
