# Quote Requests Pipeline — Incident Runbook
**Severity:** P0 — customer quote-request emails are silently being dropped
**Affected org:** Value Truck (`da3ed822-8846-4435-bb13-3cc4bf26f71d`)
**Symptom window:** ~2026-05-02 00:00 UTC → present (~58h and growing at time of writing)
**Method:** static code analysis of `main` + read-only production diagnostics. No production mutations have occurred during this analysis.

---
## 1. Overview
The `/quote-requests` page shows 0 Open Requests / 0 Auto-Captured Today, but Microsoft 365 inbound mail volume is unchanged. Static analysis of the Graph ingestion stack confirms two compounding regressions: (1) a long-standing field-extraction defect that writes empty strings when payload properties are missing, and (2) a sharp May-2 cliff where `direction='inbound'` rows effectively stopped landing in `email_messages` even though webhook + delta-sync fetch logs continue to fire and `last_sync_at` ticks normally.

The existing recovery script `scripts/recover-todays-quote-requests.ts` cannot mitigate this on its own because every recovery code path is gated on `direction='inbound'` and the database has almost none. The script must therefore be run **only after** an ingestion fix is deployed and verified. This document gives engineers the script's exact semantics, the suspected root causes to investigate, a P0 fix-task they can file as-is, and a step-by-step recovery runbook with safety gates.

## 2. Script behavior & assumptions
`scripts/recover-todays-quote-requests.ts` (551 lines) audits and recovers the quote-capture pipeline for one org.

### 2.1 Pipeline stages
| # | Stage | Reads | Writes |
|---|---|---|---|
| 1 | Mailbox + Graph subscription audit. With `--enable-disabled` flips `monitored_mailboxes.enabled=true`. Without `--no-remediate` calls `registerMailboxSubscription` for any mailbox whose subscription is missing or expires within 1h. | `monitored_mailboxes` | DB + Graph API |
| 2 | Capture leak queue: re-runs `ingestQuoteFromEmail` on every `capture_leak_reviews WHERE decision IS NULL`. | leak queue, `email_messages` | `quote_opportunities` |
| 3 | BEFORE snapshot of `quote_opportunities WHERE request_date >= CURRENT_DATE`. | — | — |
| 4 | **`backfillQuotesFromEmails(orgId, { sinceDays, useAiFallback, limit, concurrency: 5 })`** — core recovery. | `email_messages` | `quote_opportunities` |
| 5 | AFTER snapshot. | — | — |
| 6 | Live `getSnapshot()` with DEFAULT vs CLEARED filters to detect filter-hiding. | `quote_opportunities` | — |
| 7 | Parser-gate audit on today's inbound. | `email_messages` | — |
| 8 | Final report. | — | — |

NOTE: stages 1, 2, and 4 all write. There is no `--dry-run` flag at the outer script level — the only way to limit write blast radius is a combination of `--no-remediate` (skip stage 1 writes), `--no-ai` (skip the GPT-4o-mini fallback inside stage 4), and `--limit=N` (cap stage 4 row count).

### 2.2 Assumptions
- **`--org-id` is required.** No default; the script exits 2 if missing.
- **Reads are gated on `direction='inbound'`.** The decisive query at `server/services/quoteEmailIngestion.ts:1577-1580`:
  ```ts
  db.select().from(emailMessages).where(and(
    eq(emailMessages.orgId, orgId),
    eq(emailMessages.direction, "inbound"),
  ));
  ```
  Outbound rows are never read by recovery.
- **Time window.**
  - Default: strict local-midnight to now, padded by 1 minute (`hoursSinceLocalMidnightAsDays()`). Server-local clock.
  - `--since-days=N`: rolling `Date.now() - N * 86_400_000`.
  - The server is UTC; operators are MT/CT. To safely cover a same-day operator-local incident across the UTC boundary, use `--since-days=2` or higher.
- **Idempotent on `(organization_id, source='email', source_reference=providerMessageId)`** enforced inside `ingestQuoteFromEmail`. Re-running on the same input set creates 0 new rows and reports them all as `duplicates`. No double-creates are possible. Safe to interrupt and re-run.

## 3. Root-cause hypotheses (ingestion layer)
Two distinct regressions are present, with different ages and signatures.

### 3.1 Regression A — empty content on persisted rows (≥8 days old)
Production daily breakdown (Denver-local):

| Date | outbound rows | empty `from_email` | empty `subject` |
|---|---|---|---|
| Apr 27 | 3,607 | 100% | 100% |
| Apr 28 | 19,914 | 95% | 95% |
| Apr 29 | 35,827 | ≈100% | ≈100% |
| Apr 30 | 51,014 | 87% | 87% |
| May 1 | 20,640 | 85% | 85% |
| May 2 → present | every row | 100% | 100% |

Proximate code defect at `server/services/mailboxDeltaSyncService.ts:209-220` (duplicated at `mailboxHistoricalBackfillService.ts:136`):
```ts
const fromEmail = msg.from?.emailAddress?.address ?? "";
const fromName = msg.from?.emailAddress?.name ?? "";
const subject = msg.subject ?? "";
const bodyPreview = msg.bodyPreview?.slice(0, 255) ?? "";
const bodyFull = msg.body?.content ?? bodyPreview;
```
Every required field defaults to `""` when the Graph payload is missing the nested property. The persistence layer writes empties without validation. **Likely upstream trigger:** Graph delta tokens periodically return change-only payloads that omit `from`, `subject`, `body`; the `?? ""` fallback masks the failure.

### 3.2 Regression B — `direction='inbound'` stopped landing as of 2026-05-02 00:00 UTC
Cliff is sharp: last successful inbound May 1 23:59:xx UTC, May 2 00:00 onward = 0 (one stray row on May 3).

Direction inference at `server/routes/graphWebhook.ts:525-526`:
```ts
const isFromMailboxOwner = fromEmail.toLowerCase() === mailboxEmail.toLowerCase();
const direction = isFromMailboxOwner ? "outbound" : "inbound";
```
**Important inconsistency:** if `fromEmail === ""` (Regression A's symptom), this code produces `direction='inbound'`, **not 'outbound'**. The prod data — 100% outbound, 0% inbound — therefore **cannot** be explained by this path with empty `from`. A **second persistence path** must be writing the all-outbound, all-empty rows.

Candidate second paths to inspect, in likelihood order:
1. `server/routes/graphWebhook.ts:909` `processGraphNotifications` — webhook POST handler. May contain a sentitems-folder shortcut that hard-codes `direction='outbound'` and skips field hydration.
2. A SentItems-only branch within `mailboxDeltaSyncService.ingestMessage` bypassing the shared persister. (Today the code at line 188-190 differentiates only the Graph `$select`; verify no folder-conditional branching landed elsewhere.)
3. Any direct `db.insert(emailMessages).values({ direction: 'outbound', ... })` outside `processUserMailboxEmail`.
4. `server/services/conversationReplyCaptureService.ts` if recently extended to write `email_messages` directly (would only have message ID, not body).

### 3.3 Was Task #939 the cliff cause?
Commit `5e98ebcf` (Task #939 — event-driven email→quote pipeline) landed at 2026-05-01 23:59:31 UTC, **30 seconds before** the cliff. I read its full diff against `graphWebhook.ts` and `mailboxDeltaSyncService.ts`. It is **purely additive**:
- adds `messageId?: string` to `UserMailboxIngestResult`
- adds `dispatchInlineClassification({ messageId })` calls gated on `direction === "inbound" && messageId`
- reframes `emailIntelligenceScheduler` as recovery-only

It does **not** modify direction inference, field extraction, the Graph `$select`, or the persistence call. **Timing is coincidental, not causal.** More plausible triggers to rule out in this order:
1. Graph delta-token rotation around May 2 00:00 UTC returning change-only payloads
2. Azure tenant policy downgrading `Mail.Read` → `Mail.ReadBasic` (removes `body` from payloads); webhook subscriptions keep delivering under the older grant
3. A schema migration applied between 2026-04-30 22:00 UTC and 2026-05-02 02:00 UTC affecting `email_messages.{direction, from_email, subject, body}` (post-merge log already flagged `migrations/0005_carrier_intel_suggestions.sql` as conflicting state)

NOTE: do not "revert Task #939" as a reflex. Its diff is additive and unrelated; reverting would remove the inline classifier and slow inbound capture by ~2 minutes once the real fix lands, without addressing either regression.

## 4. P0 fix task (paste into tracker as-is)
> **Title:** P0: Inbound email persistence broken — Quote Requests pipeline yielding zero captures since 2026-05-02 00:00 UTC
>
> **Symptoms**
> - `/quote-requests` shows 0 Open Requests / 0 Auto-Captured today for org `da3ed822-...`
> - `email_messages` for that org: 0 rows with `direction='inbound'` for ~58h; 100% of new rows have empty `from_email`, `subject`, `body`
> - Mailbox health UI reports 32/40 healthy with sub-minute `last_sync_at`
> - `[graphWebhook] Notification accepted` and `[delta-sync] N message(s) fetched` log lines confirm Graph traffic IS arriving — the persister is dropping it on the floor
>
> **Suspected root cause** Two compounding bugs:
> 1. (Older, ≥8d) Field-extraction in the Graph delta path silently substitutes `""` (`?? ""`) when payloads omit expected nested properties
> 2. (Newer, May 2 00:00 UTC) A second persistence path exists that hard-codes `direction='outbound'` regardless of the payload, AND inbox-folder messages stopped flowing through the shared path. Likely upstream trigger: Graph delta-token rotation OR Mail.Read permission downgrade
>
> **Files / functions to inspect first** (in order)
> 1. `server/routes/graphWebhook.ts` — `processGraphNotifications` (line 909+) and `processUserMailboxEmail` (line 499). Search for any branch hard-coding `direction='outbound'` or skipping field validation.
> 2. `server/services/mailboxDeltaSyncService.ts:202-275` — `ingestMessage`. Replace `?? ""` fallbacks with explicit error+skip + alert when a required field is missing.
> 3. `server/services/conversationReplyCaptureService.ts` — confirm it does NOT write `email_messages` directly.
> 4. `server/storage.ts` — grep for `db.insert(emailMessages)` and `INSERT INTO email_messages`; confirm exactly one canonical write path exists.
> 5. `migrations/` — list anything applied between `2026-04-30 22:00 UTC` and `2026-05-02 02:00 UTC` touching `email_messages`.
> 6. Azure AD app registration — confirm Graph application permissions still include `Mail.Read` (not just `Mail.ReadBasic`).
>
> **Temporary instrumentation (revert after fix is verified)**
> ```ts
> // mailboxDeltaSyncService.ts:209 BEFORE the ?? "" fallbacks
> if (!msg.from?.emailAddress?.address || !msg.subject || !msg.body?.content) {
>   log(`[delta-sync] EMPTY-FIELD payload mailbox=${mailbox.email} folder=${folder} ` +
>       `msgId=${msg.id} hasFrom=${!!msg.from} hasSubj=${!!msg.subject} ` +
>       `hasBody=${!!msg.body?.content}`);
> }
> ```
> Repeat the probe at the top of `processUserMailboxEmail` (`graphWebhook.ts:518`). One 5-min poll cycle in production will reveal whether the Graph payload itself is empty (upstream Graph problem) vs. our extraction is wrong (code problem).
>
> NOTE: log only booleans + the Graph `messageId`, never the email body, subject, or sender address. Customer email bodies are PII; the existing `[graphWebhook]` log surface is not a sanctioned PII destination.
>
> **Acceptance criteria**
> - In a 10-min window after deploy, `email_messages` shows ≥10 rows with `direction='inbound'` AND non-empty `from_email`, `subject`, `body` for the affected org
> - `/api/internal/conversations/capture-audit-health` reports `webhookFailureCount=0`
> - Section 5 recovery runbook succeeds with `Q3 recovery delta` showing a non-zero `Δ+`
> - `/quote-requests` (non-admin rep) shows non-zero Open Requests and Auto-Captured Today
>
> **Safe deploy sequence**
> 1. Land instrumentation-only commit; deploy to prod; wait 10 min; inspect logs; **do not modify behavior yet**
> 2. Based on what the instrumentation reveals, choose one of: (a) `$select=` adjustment, (b) `fetchSingleMessage` second-pass when fields missing, (c) hard-skip + alert when fields empty, (d) Graph permission re-grant
> 3. Land fix in a small reviewable commit; deploy
> 4. Verify acceptance criteria above
> 5. Run Section 5 recovery runbook to backfill 2026-05-02 → present
> 6. Remove instrumentation in a follow-up commit
>
> **Out of scope** Tasks #967 / #968 / #969 (UI trust layer for the four ops tabs) — they operate on data already in the database and do not address ingestion correctness.

## 5. Production recovery runbook
> Execute **only after** the Section 4 ingestion fix is deployed and the acceptance criteria are met.

**Where to run:** an environment with `PRODUCTION_DATABASE_URL` set — your published Replit deployment shell, an on-call laptop with the URL pulled from your secrets manager, or your CI runner.  
**Where NOT to run:** the dev Replit workspace — its `DATABASE_URL` points at dev.

NOTE: never paste `PRODUCTION_DATABASE_URL` into a chat, ticket comment, screen-share, or shell history that is shared with anyone outside on-call. Pull it from the secrets manager into the shell that will run the script and `unset DATABASE_URL` immediately after Step 5.7.

### 5.1 Prerequisites
- Section 4 ingestion fix is deployed and acceptance criteria met
- `PRODUCTION_DATABASE_URL` is exported in the shell that will run the script
- You can `psql "$PRODUCTION_DATABASE_URL"` for verification queries
- You have notified the team in the incident channel that recovery is starting (prevents two engineers running concurrently — the script has no DB-level lock)

### 5.2 Step 1 — Sanity gate: confirm inbound is actually flowing again
```bash
psql "$PRODUCTION_DATABASE_URL" -c "
  SELECT direction,
         COUNT(*) AS rows,
         COUNT(*) FILTER (WHERE from_email != '' AND subject != '' AND body != '') AS with_content
  FROM email_messages
  WHERE org_id = 'da3ed822-8846-4435-bb13-3cc4bf26f71d'
    AND created_at >= NOW() - INTERVAL '15 minutes'
  GROUP BY direction
  ORDER BY direction;
"
```
**Stop conditions** (do NOT proceed to Step 2 if any are true):
- No `direction='inbound'` row in last 15 min → ingestion fix is not actually working
- `with_content = 0` for the inbound row(s) → fix is incomplete; field extraction still broken
- Only `outbound` rows present → the second-persistence-path bug from §3.2 is unresolved

NOTE: a single inbound row in 15 minutes is enough to satisfy the gate technically, but it is not enough to claim the fix is broadly working. Wait until you see ≥10 inbound rows with non-empty content (matching the Section 4 acceptance criterion) before running the script — otherwise you may amplify a narrow fix into a wide-blast-radius recovery against a still-broken pipeline.

### 5.3 Step 2 — Backup current state (cheap insurance)
```bash
psql "$PRODUCTION_DATABASE_URL" -c "
  SELECT COUNT(*) AS opps_before_recovery,
         COUNT(*) FILTER (WHERE outcome_status='pending') AS pending_before,
         COUNT(*) FILTER (WHERE source='email')          AS email_sourced_before,
         MAX(created_at)                                 AS most_recent_opp
  FROM quote_opportunities
  WHERE organization_id = 'da3ed822-8846-4435-bb13-3cc4bf26f71d'
    AND created_at >= NOW() - INTERVAL '4 days';
"
```
Save the output. The `Q3 recovery delta` reported by the script must match `(opps_after − opps_before_recovery)` from this snapshot.

### 5.4 Step 3 — Run the recovery script
```bash
DATABASE_URL="$PRODUCTION_DATABASE_URL" \
  npx tsx scripts/recover-todays-quote-requests.ts \
    --org-id=da3ed822-8846-4435-bb13-3cc4bf26f71d \
    --since-days=3
```
Notes:
- `--since-days=3` covers May 2 → present plus a 1-day safety margin for the UTC↔MT timezone boundary.
- Idempotent on `(organization_id, source='email', source_reference=providerMessageId)` — rows the post-fix inline classifier already captured will be reported as `duplicates`; rows from May 2 → May 3 backlog will be reported as `ingested`. Safe to re-run.
- Estimated runtime: 3–5 minutes for ~5–10k inbound emails at `concurrency=5`.
- **Recommended first run is a dry-pass:** add `--no-ai --limit=200` to bound work and skip the GPT-4o-mini fallback, eyeball the bucket samples in Section 5.5 output, then re-run without flags for the full backfill.

NOTE: `--no-ai` does **not** make the run read-only. Stage 4 still inserts `quote_opportunities` for messages the deterministic parser handles; it only skips the AI fallback for messages the parser cannot. The `--limit=200` cap above is what bounds insert volume on a first pass.  
NOTE: do **not** add `--enable-disabled` to this run. Step 5.9 is the only place that flag belongs, after the recovery itself has succeeded. Re-enabling mailboxes that were intentionally offboarded will pull mail for ex-employees.

### 5.8 Step 7 — Re-run safely if needed
If the run was interrupted or you suspect partial recovery, simply widen the window and re-run:
```bash

  npx tsx scripts/recover-todays-quote-requests.ts \
```

Newly-recovered rows will be reported as `ingested`; everything else as `duplicates`. No double-creates possible.

### 5.10 Recommended safety improvements (added to original)
- **5.1 prerequisite "notify the team"** — single-flight enforcement for human runners, since the script has no DB lock.
- **5.2 sanity gate** — three concrete stop conditions instead of "if `with_content = 0`, stop".

## 8. TL;DR for on-call
Once the Section 4 ingestion fix is deployed and you see ≥10 healthy inbound rows in production:
- Pull `PRODUCTION_DATABASE_URL` from the secrets manager into a fresh shell. Notify the incident channel before starting.
- Sanity-gate (§5.2): if inbound = 0, or `with_content = 0`, or only outbound rows present in the last 15 min — **STOP**, the fix is not actually working.

- Verify in SQL (§5.6) that May 2 / May 3 / today all show non-zero `healthy_inbound` and `email_sourced`. Verify in the app (§5.7) as both a non-admin rep and an admin.
