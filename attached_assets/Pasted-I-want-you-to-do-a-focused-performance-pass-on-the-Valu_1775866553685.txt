Sometimes the app is fine; sometimes it drags noticeably.

## Step 1 — Measure where time is going (no changes yet)

Before changing any code, profile 3 flows:

1. Load Lane Work Queue
   - Open the LWQ route from a cold start.
   - Measure:
     - Time to first byte (TTFB) for the initial API calls
     - Total page load time until lanes are visible

2. Select a lane (right-hand panel open)
   - Click a lane row.
   - Measure:
     - Time from click to panel becoming visible
     - Time from panel visible to carrier suggestions fully rendered
     - Any obvious React re-render spikes in the Profiler

3. Open History / other tabs inside the panel
   - Measure:
     - Whether tab switches trigger heavy re-fetch/re-render
     - How long until content becomes visible

Use:
- Chrome DevTools → Network + Performance

Write down concrete numbers (e.g., “initial LWQ load ~3.2s total, 1.1s TTFB, 1.4s scripting & rendering”), not just “slow” / “fast”.

## Step 2 — Identify top 3 bottlenecks

Based on these measurements, tell me exactly what the main problems are, for example:

- Backend:
  - Some API (e.g., lane list or carrier suggestions) regularly taking >500–800ms
  - Queries returning far more data than needed
  - No pagination, returning huge arrays

- Frontend:
  - Rendering hundreds/thousands of DOM nodes at once (no virtualization)
  - Heavy React state or context causing large, repeated renders
  - Large JS bundle or heavy code in the main thread on page load

- Network:
  - Overweight JSON responses (hundreds of KB or more)
  - Unnecessary parallel fetches on mount

I want you to explicitly rank them:
- Bottleneck #1
- Bottleneck #2
- Bottleneck #3

with real evidence from your profiling.

## Step 3 — Implement concrete fixes

Based on the top bottlenecks, implement **targeted** improvements. Don’t shotgun.

Here is the menu; pick and apply the ones that match the actual problems you found.

### A) Backend / API changes (if APIs are slow or heavy)

- Add or improve DB indexes on:
  - lane queries (by status, owner, org)
  - carrier suggestions (by laneId, region, equipment)
- Add pagination and/or hard limits:
  - never return more than a reasonable number of lanes or carriers in a single response
- Trim response payloads:
  - remove unused fields from the JSON (especially large nested objects)
- Add caching where safe:
  - frequently-read, slow-to-compute data should be cached server-side for a short TTL

### B) Frontend rendering changes (if React work is heavy)

- Use list virtualization for:
  - Lane lists
  - Carrier suggestion lists
  - Any other long tables/lists
- Avoid re-rendering the entire app:
  - move large data sets out of top-level context if they cause full tree re-renders
  - memoize components that don’t need to re-render on every state change
- Add query caching:
  - use React Query or similar to cache lane & carrier responses
  - avoid refetching the same lane/carrier data every time you reopen a lane
- Code-split heavy UI:
  - lazy-load the right-hand panels (carrier suggestions, history) the first time they’re opened, not on initial LWQ load
- Debounce user input:
  - search/filter boxes in LWQ should not trigger network calls on every keystroke

### C) Perceived performance / UX improvements

Even if some operations stay ~the same in raw ms, improve how it feels:

- Add skeleton loaders for:
  - lane list initial load
  - carrier suggestions
  - history tab
- Make low-risk actions optimistic:
  - marking tasks done
  - minor status changes
- Pre-fetch:
  - when a lane row is hovered or focused, start pre-loading its carrier suggestions/history so opening the panel feels instant

## Step 4 — Add tests / guardrails

Add checks so performance doesn’t slowly regress again.

Examples:

- Simple performance tests or assertions for:
  - lane list API must respond within X ms on dev data (even if approximate)
- Unit/UI tests to ensure:
  - virtualization is used for large lists (e.g., # of DOM rows <= visible window + small buffer)
  - skeletons render during initial load instead of blank or blocking UI
- Optional: a simple Lighthouse-based check or script that fails if:
  - total JS bundle or page load exceeds a certain size/time threshold

Document these thresholds in the code or README (e.g., “LWQ initial load targeted under 1.5s on a normal connection”).
