You are working inside my existing FreightDNA platform on Replit.

FreightDNA is an internal freight brokerage CRM / sales intelligence platform for helping reps capture more freight, quote faster, grow existing accounts, improve carrier coverage, and surface the next best action. It already includes CRM, AI tools, inbox/email, lane intelligence, carrier hub, available freight, lane work queue, Webex integration, SSE live sync, and multiple modular route files.

Your job is NOT to rewrite the architecture. Your job is to inspect the current system, reuse existing patterns, and improve the platform in a coordinated way so it is in a materially better spot by the end of this session.

Important constraints:
- Ask before any major architectural change.
- Reuse existing schema, routes, components, utilities, and design patterns wherever possible.
- No “as any”.
- Keep typed return values.
- Normalize all params/query access with pStr / qStr / qOptStr.
- Use existing auth, org isolation, RBAC, rate limiting, React Query, SSE, and route registration patterns.
- Do not break existing flows.
- Make incremental, production-safe improvements.
- Prefer skinny v1 implementations that create obvious user value today.
- Update replit.md at the end with the changes made, decisions, and follow-ups.

Before changing code:
1. Audit the current implementation relevant to each task.
2. Identify existing files, routes, components, hooks, tables, queries, and tests that should be reused.
3. Produce a short plan grouped by workstream.
4. Then implement.

I want you to divide the work into these 5 coordinated workstreams and execute them with side agents in parallel where appropriate.

==================================================
WORKSTREAM 1 — SPOT QUOTE SEARCH OVERHAUL
==================================================

Goal:
Make Customer Quotes / spot quote search extremely fast, intuitive, and useful for winning more freight.

Build or improve a top-of-page spot quote command bar that asks for:
- Pickup city/state
- Delivery city/state
- Mode (Van, Reefer, Flatbed, Other)
- Pickup date

Optional advanced filters can live in an expander.

The result experience should prioritize:
- Exact lane history
- Similar lane history
- Won vs lost quote history
- Paid carrier rate history
- Internal quote range / average / median where available
- Market band references where available
- Confidence score
- Staleness indicator
- “Why this recommendation” explanation
- Suggested next actions such as quote now, follow up, carrier outreach, open lane detail

UX requirements:
- Search bar pinned at top of the tab or page
- Extremely keyboard-friendly
- Fast-loading
- Clean result grouping
- Swap pickup/delivery shortcut
- Strong empty states, loading states, and error states
- No clutter; prioritize speed to answer

Implementation guidance:
- Reuse existing customer quotes, load_fact, won/lost, TRAC/SONAR, carrier history, and lane intelligence logic where possible
- Do not invent fake data
- If data is missing, clearly label it
- Keep API and UI modular

Deliverables:
- Improved UI
- Supporting query/service logic
- Any needed route additions or route cleanup
- Tests for search behavior and edge cases

==================================================
WORKSTREAM 2 — FREIGHT CAPTURE FUNNEL
==================================================

Goal:
Create a closed-loop funnel so we can see where freight is leaking out of the process.

Build a simple but useful freight capture funnel that tracks, at minimum:
- Request received
- Quoted
- Follow-up sent
- Booked / won
- Lost
- Stale / no response
- Reason lost (when known)

Slice the funnel by:
- Rep
- Customer
- Company
- Lane
- Mode
- Time period

Outputs should include:
- Quote-to-book conversion
- Average response time
- Follow-up compliance
- Top lost reasons
- Stale quote counts
- Best and worst performing lanes/customers/reps

Implementation guidance:
- Reuse existing touchpoints, email/inbox, quote, lane, and activity/event structures if possible
- If there is no single canonical event stream, create a pragmatic v1 event model rather than a giant rewrite
- Make it easy to expand later
- Surface this in an internal dashboard view, card set, or focused tab

Deliverables:
- Event/status model
- Backend logic and route support
- Visible reporting UI
- Clear definitions of each funnel stage
- Tests for stage transitions and metrics logic

==================================================
WORKSTREAM 3 — DAILY PRIORITIES / NEXT BEST ACTION WORKSPACE
==================================================

Goal:
Give reps one place to start the day with the clearest ranked actions instead of making them hunt through the app.

Build or improve a daily priorities workspace that groups work into:
- Quote Now
- Follow Up
- Defend
- Grow
- Procure Carrier

Each card/task should show:
- Why it surfaced
- Priority / urgency
- Expected value or impact signal if feasible
- Recommended action
- Deep link to the correct workflow
- Owner and due timing

Logic ideas:
- Quote requests waiting too long
- High-potential accounts with low share of wallet
- Recent missed calls or unreturned conversations
- At-risk customers with declining engagement
- Recurring lanes with weak carrier depth
- New inbox signals needing action
- Lanes with recent wins but low follow-up or low penetration

Implementation guidance:
- Reuse existing NBA engine and rule patterns where possible
- Prefer a single clear action over too many weak suggestions
- Do not flood the user with noise
- Rank and dedupe aggressively

Deliverables:
- Prioritized daily workspace UI
- Rule/scoring logic
- Deep-linking into core workflows
- Tests proving cards fire correctly and do not duplicate excessively

==================================================
WORKSTREAM 4 — RECURRING LANE PROGRAM BUILDER
==================================================

Goal:
Help NAMs, AMs, and LMs work freight we already have in house and line up more carrier coverage on lanes that matter.

Build a v1 recurring lane workflow for lanes with consistent freight, using a configurable threshold such as 2+ loads/week or equivalent over a lookback period.

For each qualifying lane, show:
- Why the lane qualifies as recurring
- Customer / lane / mode
- Volume / recency trend
- Existing carrier depth
- Recommended carriers from history
- Outreach readiness
- Gaps in coverage
- Next action

Support actions like:
- View lane detail
- Open carrier suggestions
- Generate outreach email draft
- Launch outreach workflow
- Mark lane as in progress / covered / needs attention

Implementation guidance:
- Reuse load history, financial uploads, load_fact, carrier hub, email, and outreach logic where possible
- Prefer source-linked, explainable recommendations
- If AI is used, keep outputs reviewable and grounded in existing data
- Do not auto-update carrier profiles from inferred data without auditability

Deliverables:
- Detection logic for recurring lanes
- Lane detail / suggestions UI
- Carrier recommendation support
- Outreach integration hooks
- Tests for recurring-lane qualification and recommendations
