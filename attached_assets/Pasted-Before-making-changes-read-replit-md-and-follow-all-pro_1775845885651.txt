Before making changes, read replit.md and follow all project guardrails.

We are now designing the shared middle layer for a new intelligence workflow in the Value Truck app.

This is the common engine that will sit between:
1. inbound demand/capacity events
and
2. downstream next-best-action workflows on both the customer side and the carrier side

Do NOT start by building customer UI or carrier UI first.
Start by building the shared Market Signal layer so both sides can use the same underlying signal model, thresholds, evidence, and audit trail.

Business goal:
Turn raw operational activity into explainable, actionable market signals.
Examples:
- spot quote requests from GA/FL are rising
- reefer demand in the Southeast is heating up
- outbound FL dry van requests are above baseline
- carrier-declared interest in a region is rising
- demand is increasing faster than known available carrier coverage

These signals will later drive:
- customer-side NBAs (call exposed shippers proactively)
- carrier-side NBAs (source/add capacity proactively)
- ranking and planning workflows

Important product principle:
This must be explainable, threshold-driven, and auditable.
Do NOT build a black-box AI alerting system.
Use clear business rules first, with room for future ML/anomaly scoring later.

What to build in v1:
Build the shared Market Signal object, detection pipeline, thresholds, persistence, and minimal internal visibility needed to support future workflows.

Scope:
1. event inputs
2. market signal data model
3. threshold detection logic
4. signal lifecycle logic
5. evidence / explainability payload
6. minimal API/service surface
7. tests

Core concept:
Raw events should not directly create NBAs.
Raw events should roll up into Market Signals.
Then Market Signals can fan out into downstream actions.

--------------------------------------------------
1. EVENT INPUTS
--------------------------------------------------

Design the Market Signal engine to consume normalized internal events.
Do not assume only one source forever.

Support at least these event types in the model/service design:
- customer_demand_event
- carrier_capacity_event
- carrier_declared_lane_event
- customer_quote_request_event
- procurement_outreach_response_event

For v1, it is okay if only a subset is actually wired in, but design the data model and service boundaries so multiple event types are supported.

At minimum, each normalized input event should be able to carry:
- eventType
- eventTimestamp
- sourceEntityType (account, carrier, lane, email, quote, etc.)
- sourceEntityId
- accountId if applicable
- carrierId if applicable
- laneId if applicable
- originRaw
- destinationRaw
- originRegion / destinationRegion if available
- equipmentType if available
- loadMode if relevant
- confidence if event extraction is AI-assisted
- metadata JSON for raw context/evidence

If existing lane normalization utilities already exist, reuse them.
Do not invent a second normalization system if one already exists.

--------------------------------------------------
2. MARKET SIGNAL OBJECT
--------------------------------------------------

Create a durable Market Signal model/table/object.
This is the key shared middle-layer object.

A Market Signal should represent a meaningful, aggregated operating condition, not a single event.

At minimum support these fields:
- id
- signalType
- marketScopeType
- marketScopeKey
- originRegion
- destinationRegion
- equipmentType
- loadMode
- status (active, cooling, resolved, suppressed)
- severityLevel (info, watch, elevated, urgent)
- signalScore (0-100)
- thresholdVersion
- windowStart
- windowEnd
- detectedAt
- lastEvaluatedAt
- resolvedAt nullable
- supportingEventCount
- baselineEventCount nullable
- percentChange nullable
- evidenceSummary JSON
- createdBySystem boolean
- suppressionReason nullable
- metadata JSON

Recommended signalType enum examples:
- demand_surge
- quote_surge
- regional_heat
- equipment_heat
- carrier_capacity_rise
- carrier_capacity_gap
- demand_capacity_imbalance

Recommended marketScopeType examples:
- region
- corridor
- origin_region
- destination_region
- equipment_region
- lane_cluster

Examples:
- signalType=demand_surge, marketScopeType=origin_region, marketScopeKey=GA, equipmentType=dry_van
- signalType=demand_capacity_imbalance, marketScopeType=corridor, marketScopeKey=GA->FL|reefer

Important:
Signals must be durable objects with a lifecycle.
Do NOT treat them as ephemeral computed-only values with no persistence.

--------------------------------------------------
3. THRESHOLD LOGIC
--------------------------------------------------

Implement v1 using explicit threshold logic, not ML.

Use a hybrid approach:
- absolute minimum event count floor
- relative change vs baseline
- optional recency acceleration check
- confidence floor if source events are AI-extracted

Example philosophy:
A region should not “heat up” because of one or two noisy events.
A signal should require enough volume and enough change.

Please implement a configurable threshold system.

Recommended threshold config structure:
- minEventCount
- baselineLookbackDays
- evaluationWindowHours
- minPercentIncrease
- minimumDistinctAccounts or carriers
- minimumDistinctSourceThreads if email-derived
- coolDownHours
- autoResolveAfterHours
- severity cutoffs
- confidenceFloor if applicable

Recommended starter thresholds for v1:
1. demand / quote surge
- evaluation window: 72 hours
- baseline lookback: trailing 14 days
- minimum events in window: 5
- minimum percent increase vs normalized baseline: 35%
- minimum distinct accounts: 2
- urgent threshold if >= 60% above baseline and >= 8 events

2. carrier declared lane / capacity rise
- evaluation window: 7 days
- baseline lookback: trailing 30 days
- minimum events: 4
- minimum distinct carriers: 3

3. demand-capacity imbalance
- if demand_surge exists in a market and matching carrier capacity signals are weak or absent, create imbalance signal
- this should be explainable from component signals, not magic logic

Use simple, readable formulas.
If baseline is zero or near zero, use guardrails so we do not create absurd infinite-growth signals from tiny counts.
Examples:
- require absolute count floor before a zero-baseline market can signal
- optionally use a fallback severity cap when baseline is too sparse

--------------------------------------------------
4. SIGNAL LIFECYCLE
--------------------------------------------------

Signals need lifecycle logic.

Implement at least:
- active: threshold currently met
- cooling: previously active, now falling below threshold but still recent
- resolved: no longer active after cooldown / evaluation window
- suppressed: manually or system-suppressed

Behavior:
- signal opens when thresholds are met
- signal remains active while still supported
- signal transitions to cooling when no longer strongly supported but still within relevance window
- signal auto-resolves after inactivity / cooldown
- repeated evaluations should update the same active signal when the scope/type matches, not create duplicates every run

Important:
Deduplicate by active signal identity:
- signalType
- scope
- equipment
- active window overlap / current status

Do not create dozens of duplicate active signals for the same GA dry van heat pattern.

--------------------------------------------------
5. EVIDENCE / EXPLAINABILITY
--------------------------------------------------

Every signal must be explainable in plain language and supported by evidence.

Store evidence summary data such as:
- current event count
- baseline event count
- percent increase
- distinct accounts / carriers
- top affected regions or corridors
- sample supporting source ids
- top equipment types
- last N relevant events references

Add a generated explanation field or helper such as:
- “Outbound GA dry van quote activity is 42% above its trailing 14-day baseline, with 9 requests across 4 accounts in the last 72 hours.”

Important:
This explanation should be deterministic from the data.
Do not generate it with an LLM.

Downstream users need to trust why a signal exists.

Add a minimal internal service/API layer.

Need:
- a service to ingest normalized events into the signal evaluation pipeline
- a service to evaluate/re-evaluate signals for a scope
- a way to fetch active signals
- a way to fetch signal details/evidence

If there is already a pattern for internal services/jobs, follow it.

Suggested capabilities:

It is okay if v1 only exposes internal endpoints or service functions.
We do not need polished user-facing UI yet.

--------------------------------------------------
7. DESIGN RULES
--------------------------------------------------

Important implementation rules:

- Keep market signals separate from NBAs.
  Signals are shared intelligence; NBAs are downstream actions.

- Keep signals separate from raw events.
  Raw events are evidence, not the signal itself.

- Reuse existing normalization utilities for lanes/regions/equipment if they already exist.

- Keep all threshold values configurable in one place.
  Do not scatter magic numbers across files.

- Make the signal engine explainable and testable.
  We should be able to show exactly why a signal fired.

- Favor false negatives over noisy false positives in v1.
  This system should earn trust before it becomes aggressive.

- Preserve source attribution for any AI-assisted event extraction.

- Do not build customer-facing or carrier-facing outreach automation in this PR unless a tiny placeholder is needed for wiring.

--------------------------------------------------
8. TESTS
--------------------------------------------------

Add/update tests for at least:

Threshold behavior
- does not create signal for tiny/noisy event counts
- creates demand surge when count and percent increase thresholds are met
- does not create surge if count is high but distinct-account floor is not met
- handles sparse/zero baseline safely

Lifecycle behavior
- updates existing active signal instead of duplicating it
- transitions active -> cooling -> resolved appropriately
- respects cooldown / auto-resolve behavior

Evidence behavior
- stores supporting counts and percent change correctly
- explanation text matches the computed evidence

Scope behavior
- distinguishes region vs corridor vs equipment-region signals correctly
- does not merge unrelated equipment types into one signal

Imbalance behavior
- creates demand-capacity imbalance only when demand signal is active and capacity support is insufficient

Regression / safety
- no duplicate active signals for same scope/type
- config-driven thresholds work as expected
