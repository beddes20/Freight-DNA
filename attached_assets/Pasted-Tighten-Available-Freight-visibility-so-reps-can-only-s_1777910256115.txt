Tighten Available Freight visibility so reps can only see their own freight when “viewing as” a specific user.

Current behavior:
- On the Available Freight cockpit, when I choose “You are viewing as Jared Reynolds (national account manager)”, the page still shows freight for customers that are NOT on Jared’s book or team (see highlighted rows in the current UI — they belong to other teams/customers). [file:31]
- Owner / Ops User multi-select and buckets are wired up and working, but the initial data scope is too broad; the page starts from a global queue and then layers filters.

Product requirement:
When I am in “viewing as <rep>” mode for Available Freight:
1) **Default scope must be “only this rep’s freight”**, not the global queue.
2) Optionally, we can allow promotion to “my team’s freight”, but that should be an *explicit* choice (e.g., via Owner filter), not the default.
3) Reps should NEVER see freight for customers that are completely outside their ownership scope when they are viewing as themselves or as another individual rep.

Implementation details and constraints:
- Use the existing owner / team infrastructure (shared/cockpitTeams.ts, cockpitTeamMap.json, resolveOwnerScope, and the Owner combobox that already supports me / my-team / team:<id> / specific user / unassigned).
- The **server route** (server/routes/freightOpportunityCockpit.ts) must:
  - Detect when the request is in “view as <rep>” mode (however currentIdentity is passed: impersonatedUserId, viewingAsUserId, etc.).
  - Apply a **hard server-side owner scope** based on that user BEFORE any client filters:
    - If viewing as Jared, base scope = { Jared’s userId } (plus optionally his direct reports if “my team” is our business rule).
    - Only rows whose ownership/ownerId maps into that scope should be returned in the feed.
    - Everyone else’s freight should be completely excluded from `feed.items` and from server KPIs.
- The **client** (available-freight.tsx) should:
  - Default the Owner filter to “Me” when viewing as that rep.
  - Never silently widen owner scope on initial load beyond the viewing-as user’s allowed scope.
  - Allow explicit expansion (e.g., Me + My Team) only if we intentionally want that for some roles.
  - Make sure `applyCockpitFilters` never re-introduces rows outside the server owner scope (it should only narrow further).

Specific requests:
1) Server-side enforcement
- In freightOpportunityCockpit.ts, add a step early in the query construction that:
  - Reads the effective user identity (including “view as”).
  - Resolves a **baseOwnerScope** via resolveOwnerScope using tokens like ["<viewingAsUserId>"] or ["team:<id>"] if our rule for Jared is “his team”.
  - Applies this baseOwnerScope predicate directly in the DB query so only matching rows are returned from the server.
- This baseOwnerScope must be **non-optional** when in a “view as <rep>” context.
- Add tests that:
  - When viewing as Jared, rows whose customer/owner are clearly outside Jared’s book never appear in server results.
  - When not viewing as anyone (admin/global mode), existing behavior remains unchanged.

2) Client defaults
- In available-freight.tsx, make the Owner combobox default selection reflect the viewing-as user:
  - If impersonating Jared, default Owner to “Me” (which will be Jared in that context).
  - Ensure the active filter chips and URL (?owner=…) show this explicitly.
- Ensure that clearing filters via “Reset to operational default” does NOT remove the base owner scope derived from viewing-as; it should reset to “Me” again, not to All.

3) Guardrails
- Add a small internal debug mode (e.g., ?debug=cockpit) log that prints:
  - currentIdentity (including viewing-as id),
  - baseOwnerScope (server),
  - effective ownerFilter tokens (client),
  - counts of rows per owner at the server boundary.
- If any row makes it to the client that does NOT satisfy baseOwnerScope while in viewing-as mode, log a loud warning in dev (?debug=cockpit) so we can catch future leaks.

4) Tests
- Extend cockpitTeams / cockpitFilters tests and the standalone cockpit-hardening harness to cover:
  - viewing-as scope: given a viewingAsUserId, baseOwnerScope only includes that user (and optional team mapping).
  - “Reset to operational default” in viewing-as mode preserves the base owner scope.
- Add at least one server-side test that simulates viewing as a rep and verifies that items with an unrelated owner never appear.

5) Non‑goals
- Do NOT change the bucket logic, pickup rules, or KPI-from-filtered behavior that we just hardened.
- Do NOT expand this scope to other pages; this task is for the Available Freight cockpit only.

Goal:
After this change, when I am viewing as Jared Reynolds in Available Freight:
- Every row on the page is Jared’s freight (or his team’s, if that’s our explicit rule).
- I do NOT see freight for accounts that are outside his book unless I intentionally widen via an allowed owner filter in a non-viewing-as context.
