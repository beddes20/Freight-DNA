2) Tier 1 — “Stop wasting trust” (do these first)

Implement these as sub‑tasks under this one mission:

2.1 Bounce / DSN / Out‑of‑Office classifier

Add robust detection for:

hard bounces / invalid addresses,

soft bounces / mailbox full,

out‑of‑office / auto‑reply patterns.

Store results in a durable structure (e.g. email_bounce_events or similar) linked to:

contact,

thread,

and message.

Behavioral requirements:

Suppress NBAs and automated follow‑ups to clearly dead or OOO contacts.

Surface a clear signal so reps and managers can see which contacts are bouncing or out.

Ensure manual rep‑sent emails still go out, but surface a warning where helpful.

Add tests and a small diagnostic script that can summarize bounce/OOO counts per day and per contact.

2.2 Email participant explosion (CC/BCC graph + stakeholder change detection)

Stop treating to/cc as raw strings only. Introduce an email_participants model:

one row per unique email address per thread (or per message if necessary),

roles: sender, to, cc, bcc, original sender on forwards.

Build a stakeholder graph per account based on this:

active participants,

silent observers (cc’d repeatedly but rarely write),

churn detection (previously active participants who go quiet for N days).

Expose this in a minimal way (data + one or two internal views). It does not need a fancy UI yet.

This must be implemented in a way that supports later NBAs like “new stakeholder appeared” and “stakeholder churn risk.”

2.3 Attachment router (especially rate cons) → facts

Extend attachment handling beyond PODs and truck lists:

classify attachments into: POD, rate con, BOL, COI, MSA/contract, RFP workbook, generic.

Focus implementation on rate cons first.

For rate cons:

Extract lane, rate, dates, and relevant contract terms into a structured form.

Write back to load / lane rate evidence where appropriate, so pricing can learn from actual awarded rates.

Keep the router modular so other attachment types can be wired later.

3) Tier 2 — Signal‑to‑fact crystallization (build these next)

Once Tier 1 is solid, implement these under the same mission:

3.1 Slot extractor for competitive and timing facts

Add structured extraction for:

targetRate, incumbent, incumbentRate, competitorName,

rfpDate, contractEndDate,

core lane attributes (equipment, commodity, weight, temp, transitDays) where not already captured.

Write these into appropriate tables:

competitive_signals (wired from email, not just stubbed),

lane / rate history evidence tables,

a simple forward_calendar structure for RFP/contract dates.

Wire them so that:

lost/won outcomes enrich lane rate history with competitor/rate context,

RFP/contract dates become future NBAs (prep, renewal risk, etc.).

3.2 Promise register

Detect when a rep promises something in email (especially time‑bound commitments: “by EOD Thursday,” “tomorrow morning,” “this afternoon”).

Store commitments as structured promises:

who promised (rep),

to whom (contact/account/carrier),

what was promised (short description),

due time,

status (kept, broken, unknown).

Implement a basic mechanism to mark promises as kept/broken based on subsequent thread events (reply sent, load booked, etc.), even if it’s imperfect at first.

Keep scope tight: this is a backend fact + a basic way to surface it for coaching; no full UI redesign needed.

3.3 Question register

Detect explicit questions customers ask in email threads.

Track whether each question receives a corresponding answer from the rep.

Store:

question text / category,

asked time,

answered/unanswered status,

time to answer.

This is the foundation for “question‑answered rate” coaching metrics.

3.4 Outbound quality grader

For every rep‑sent email, compute lightweight quality scores:

clarity, tone, value‑add, and objection handling (where applicable).

Store scores in a way that can be rolled up per rep and per account.

Do not block sends; this is diagnostic only.

Calibrate initially with heuristic thresholds; we can refine later based on outcomes.

3.5 Sentiment writeback

Populate contact_sentiment_tracking (or equivalent) from email threads:

per‑message sentiment,

per‑contact sentiment trend,

basic smoothing over time.

Ensure sentiment is queryable for later Account Health scoring and NBAs.
