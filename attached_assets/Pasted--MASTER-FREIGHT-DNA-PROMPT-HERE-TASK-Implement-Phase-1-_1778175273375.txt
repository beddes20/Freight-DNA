Phase 1 Step 3 — Lifecycle Write Paths: Plan

1. Diagnosis
Phase 1 has the schema (13 lifecycle columns + user_lifecycle_events table) and pure helpers (server/lib/userLifecycle.ts), but no way to actually move a user between states. Today the only mutator is DELETE /api/users/:id (hard-delete-style) and PATCH /api/users/:id (general profile patch). Neither writes audit events. Roster Health (Phase 0) shows admins what's wrong but gives them no button to act.

What's missing for Step 3:
- Five write methods on IStorage that flip lifecycle columns + emit a user_lifecycle_events row in the same transaction.
- Five admin-only POST routes + one read route for the audit log + one read route for blast-radius preview.
- Server-side rules that encode the policy in the helper module's docstrings (real-employee-leaving → deactivate; junk → soft-delete; restore → returns to inactive; service accounts can't be flipped interactive by accident).
- No call-site reads change. Auth, GET /api/users, dashboards, goals, quotes, NBA, Webex, Stripe, contact jobs, assignee dropdowns — none of them yet read these flags. That migration is Step 4+ and explicitly out of scope here.

2. What is trustworthy vs not trustworthy

Trustworthy now (won't shift under us):
- users lifecycle columns + their defaults (Step 1, migration 0017).
- user_lifecycle_events table (Step 1, migration 0018) — inserts only, append-only contract.
- server/lib/userLifecycle.ts predicates — pure, unit-tested, not yet read anywhere, so writes can land before reads catch up without behavior drift.

- requireAuth + isAdmin (server/lib/roles.ts) + getCurrentUser (server/auth.ts) — same admin gate pattern used by adminRosterHealth.ts and adminEmailDerivedCompanies.ts.
- Existing searchUsers projection (Step 1 + Step 2 fix) — already includes the new lifecycle columns.

Not trustworthy / untouched on purpose:
- DELETE /api/users/:id (server/routes.ts:1158) — currently does a hard delete with cascade-style cleanup; leave it alone for this step. Rewriting it to soft-delete is a routing-contract change that downstream UIs depend on.
- PATCH /api/users/:id (server/routes.ts:1052) — general profile patch; lifecycle fields are already omitted from insertUserSchema so the patch path can't accidentally flip them. Don't widen it.
- POST /api/users and POST /api/users/bulk-import — these create rows that will land with default flags (isActive=true, others=false). That's intentional; classification happens after-the-fact via the new admin route.
- Default GET /api/users filter — explicitly NOT changed in this step (the helper docstring at isVisibleInDefaultRoster already calls this out).
- users.id PK type is varchar and existing routes like /api/users/:id use param-as-string — admin lifecycle routes will follow the same pattern.

Routes (all admin-only, org-scoped, return 403 when caller isn't isAdmin or target is in another org):

- POST /api/admin/users/:id/classify
  Body: { isServiceAccount?, isDemo?, isFixture?, isQuarantined?, reason? }
  Effect: Sets non-destructive classification flags. Refuses to set isServiceAccount=true on a row with isActive=true && deletedAt IS NULL && isDemo=false && isFixture=false unless body also sets isActive:false (the "service account never silently consumes a seat" guard).
  Audit event: classify

- POST /api/admin/users/:id/deactivate
  Body: { reason } (required)
  Effect: isActive=false, deactivatedAt=now(), deactivatedBy=actor, deactivationReason=reason. No-op (idempotent 200) if already inactive — emits no duplicate audit row.
  Audit event: deactivate

- POST /api/admin/users/:id/reactivate
  Body: { reason? }
  Effect: isActive=true, clears deactivatedAt/By/Reason. Refuses if deletedAt IS NOT NULL (must restore first). Refuses if isServiceAccount=true (admin must explicitly clear that flag via classify first).
  Audit event: reactivate

- POST /api/admin/users/:id/soft-delete
  Body: { reason } (required)
  Effect: deletedAt=now(), deletedBy=actor, deleteReason=reason, also isActive=false. Mirrors the contacts soft-delete pattern from Task #1093. Refuses if user has any open downstream ownership (companies.assignedTo / ownerRepId / salesPersonId, opportunities, tasks); the impact endpoint is the pre-flight check. Override via ?force=true query param (still audited, the audit row carries force: true in nextState).
  Audit event: soft_delete

- POST /api/admin/users/:id/restore
  Body: { reason? }
  Effect: Clears deletedAt/By/Reason. Per the rule, leaves isActive=false — restore returns the user to inactive, not active. Caller must follow up with reactivate if they want a live employee.
  Audit event: restore

- GET /api/admin/users/:id/lifecycle-events
  Body: —
  Effect: Org-scoped read of user_lifecycle_events for that user, newest first, with actor name resolution.

- GET /api/admin/users/:id/impact
  Body: —
  Effect: Read-only blast radius: counts of owned companies (assignedTo + ownerRepId + salesPersonId), open opportunities, open tasks, last 30 days of touchpoints, and lastActivityAt. Returns the softDeleteWouldBlock: boolean flag the soft-delete route checks.

5. Risks / dependencies

Risk → Mitigation:
- Someone wires these routes into the admin UI before reads are migrated, then admins flip isActive=false on a real rep and nothing changes (because dropdowns/dashboards still show them).
  → Keep the admin UI work explicitly out of this task. Replit gotcha entry says "Step 3 ships server-side; UI lands in Step 5 alongside read-side filter migration."

- Soft-delete cascading away ownership invisibly.
  → Don't cascade. soft-delete only flips the user row; ownership rebinding is Step 4+ (a deliberate, separate migration). The impact endpoint surfaces the count so admins know what they're leaving orphaned. The ?force=true override puts the burden on the admin and audits it.

- DELETE /api/users/:id continues to hard-delete and bypass the audit log.
  → Out of scope. Replacing that route's contract is Step 6 (after read-side migration) — flagged in §7.

- Bulk-import creates rows pre-classified incorrectly.
  → Out of scope; bulk-import already uses default flags. Admins can post-classify via the new route.

- Race between two admins acting on the same row.
  → SELECT … FOR UPDATE inside the transaction serializes. Idempotent no-ops keep the audit log clean.

- Service account silently regaining interactive login.
  → reactivateUser refuses when isServiceAccount=true; admin must classify {isServiceAccount: false} first. Encoded in storage method, not just route layer, so future callers can't bypass it.

- Org-scope bypass (admin in org A acting on user in org B).
  → Storage method takes actor.organizationId and 404s on mismatch — checked at storage layer, not just route layer.

- Audit row missing on partial failure.
  → UPDATE + INSERT in the same db.transaction; either both land or neither does.

- Phase 1 step 1 migration (0017/0018) hasn't been pushed in some environment.
  → Storage methods will fail loudly on missing column — acceptable; production deploy already requires drizzle-kit push:pg.

- Cross-org or platform-admin variants. Only same-org admin can act; no super-admin escape hatch.
