Lane Pricing & History — Final Implementation Specification
A single-page internal web app for freight brokers to record and look up the quotes they've previously submitted to customers on a given lane, with placeholders for live external market data, a 12-week lane forecast, and McLeod load-history integration.
This document is self-contained: another developer (or another Replit Agent) should be able to recreate the same app from this spec alone.
________________________________________
1. Tech Stack
•	Monorepo: pnpm workspaces.
•	Frontend: React 18 + Vite + TypeScript, TailwindCSS, shadcn/ui components (Radix primitives), react-hook-form + zod for the quote form, @tanstack/react-query for data fetching, lucide-react icons, date-fns for date formatting. The OpenAPI client is generated with @hey-api/openapi-ts and consumed via TanStack Query hooks.
•	Backend: Node.js + Express + TypeScript. Routes are defined per resource (/me, /locations, /lanes/history, /quotes, /healthz). Request/response shapes are validated with Zod schemas generated from the shared OpenAPI document.
•	Database: PostgreSQL accessed through Drizzle ORM (drizzle-orm + drizzle-kit). Connection string read from DATABASE_URL. Schema is pushed with drizzle-kit push (no manual SQL migrations).
•	Auth: Cookie-based, signed with HMAC-SHA256 using SESSION_SECRET. No external IdP. The "username" is whatever the user typed in the welcome dialog.
•	Shared API contract: A single openapi.yaml lives in a shared workspace package. Both server (Zod validators) and client (typed hooks) are generated from it.

3. Database Schema
All tables live in the default schema. IDs are serial (auto-increment integers). Money/mileage values use numeric (Postgres arbitrary precision) and are serialized to/from numbers in the API layer. created_at defaults to now().
3.1 lanes
A normalized lane definition. One row per (origin, destination, equipment).
Column	Type	Nullable	Default	Notes
id	serial	no		PK
origin_city	text	no		
origin_state	text	no		2-letter
dest_city	text	no		
dest_state	text	no		2-letter
equipment	text	no		One of: VAN, REEFER, FLATBED
miles	numeric	yes	null	Cached lane mileage; first quote that supplies miles fills it in.
Unique index lanes_unique_idx on (origin_city, origin_state, dest_city, dest_state, equipment).
3.2 quotes
Every quote our team has submitted to a customer.
Column	Type	Nullable	Default	Notes
id	serial	no		PK
lane_id	integer	no		FK → lanes.id, ON DELETE CASCADE
customer_name	text	no		Required
all_in_amount	numeric	no		Required, the headline quote
miles	numeric	yes	null	Snapshot of lane miles at submission time
fsc_rpm	numeric	yes	null	Fuel surcharge per mile, in $
fsc_total	numeric	yes	null	miles * fsc_rpm (computed server-side)
linehaul_amount	numeric	yes	null	all_in_amount - fsc_total (computed server-side)
notes	text	yes	null	
created_at	timestamp with time zone	no	now()	Auto-filled
created_by	text	no		Auto-filled from session cookie
3.3 loads
Reserved for the McLeod integration (currently unused; the bottom panel shows "Coming soon").
Column	Type	Nullable
id	serial	no
lane_id	integer	no (FK → lanes.id, cascade)
customer_name	text	yes
run_date	date	yes
ops_owner	text	yes
freight_charges	numeric	yes
other_charges	numeric	yes
total_charges	numeric	yes
source_system	text	yes
3.4 cities
Reserved for a future DB-backed city autocomplete; the current implementation reads from a hard-coded CITIES list of US city/state pairs on the server (~350 entries).
Column	Type	Nullable
id	serial	no
city	text	no
state	text	no

4. API
All routes are mounted under /api. Request bodies are JSON. Responses are JSON. Errors return { "error": string } with an appropriate 4xx/5xx status. The session cookie lm_user is httpOnly + signed.
4.1 GET /api/healthz
Liveness probe.
Response:
{ "status": "ok" }
4.2 GET /api/me
Returns the currently signed-in user (or null if not signed in).
Response:
{ "username": "JS" }
or
{ "username": null }
4.3 POST /api/me
Sets the signed cookie identifying the current user. There is no password — this is an internal tool; the username is just typed in.
Body:
{ "username": "JS" }
Response:
{ "username": "JS" }
Sets cookie lm_user (httpOnly, sameSite=lax, 90-day max-age).
4.4 GET /api/locations?query=<text>
Autocomplete. Case-insensitive substring match against the curated city list. Returns up to 10 results. Empty query → [].
Example: GET /api/locations?query=dal
Response:
[
  { "city": "Dallas", "state": "TX" },
  { "city": "Dalton", "state": "GA" }
]
4.5 GET /api/lanes/history
Find or create the lane for the given (origin, dest, equipment) and return its historical quotes (most recent first, capped at 20).
Query params (all required):
•	origin_city, origin_state
•	dest_city, dest_state
•	equipment — one of VAN, REEFER, FLATBED (case-insensitive)
Example:
GET /api/lanes/history?origin_city=Dallas&origin_state=TX&dest_city=Chicago&dest_state=IL&equipment=VAN
Response:
{
  "lane": {
    "id": 1,
    "origin_city": "Dallas",
    "origin_state": "TX",
    "dest_city": "Chicago",
    "dest_state": "IL",
    "equipment": "VAN",
    "miles": 962
  },
  "last_quote": {
    "id": 14,
    "lane_id": 1,
    "customer_name": "Acme Logistics",
    "all_in_amount": 2070,
    "miles": 962,
    "fsc_rpm": 0.78,
    "fsc_total": 750.36,
    "linehaul_amount": 1319.64,
    "notes": null,
    "created_at": "2026-04-23T15:02:11.000Z",
    "created_by": "JS"
  },
  "quotes": [ /* up to 20 Quote objects, newest first */ ],
  "loads": []
}
4.6 POST /api/quotes
Create a quote. Requires the lm_user cookie.
Body:
{
  "origin_city": "Dallas",
  "origin_state": "TX",
  "dest_city": "Chicago",
  "dest_state": "IL",
  "equipment": "VAN",
  "customer_name": "Acme Logistics",
  "all_in_amount": 2500,
  "miles": 962,
  "fsc_rpm": 0.78,
  "notes": "Drop-and-hook on origin"
}
Required: origin_city, origin_state, dest_city, dest_state, equipment, customer_name, all_in_amount. All others optional.
Server behavior:
1.	Reject if no signed-in user (HTTP 401).
2.	Validate equipment is one of VAN/REEFER/FLATBED.
3.	Find or create the lane row.
4.	Resolve effective miles: prefer body.miles, else fall back to lane.miles.
5.	Compute fuel helper if both miles and fsc_rpm are present:
•	fsc_total = round(miles * fsc_rpm, 2)
•	linehaul_amount = round(all_in_amount - fsc_total, 2)
6.	Insert the quote with created_by = currentUser and created_at = now() (auto).
7.	If the lane had no miles and the quote supplied them, update the lane so future history responses pre-fill miles.
Response:
{
  "quote": {
    "id": 15,
    "lane_id": 1,
    "customer_name": "Acme Logistics",
    "all_in_amount": 2500,
    "miles": 962,
    "fsc_rpm": 0.78,
    "fsc_total": 750.36,
    "linehaul_amount": 1749.64,
    "notes": "Drop-and-hook on origin",
    "created_at": "2026-04-23T15:30:00.000Z",
    "created_by": "JS"
  }
}

5. Authentication & created_by
•	The first time a browser hits the page, a modal "Welcome to Lane Pricing & History" asks for a name or initials.
•	POST /api/me { username } writes a signed cookie lm_user (username.<HMAC-SHA256(SESSION_SECRET, username)>, httpOnly, sameSite=lax, 90-day expiry).
•	All subsequent requests carry that cookie. GET /api/me returns the decoded username so the UI can show a "Signed in as X — Change" badge.
•	POST /api/quotes reads the cookie, decodes/validates the HMAC, and uses the resulting username as created_by. If the cookie is missing or tampered with, the server returns 401.
•	There are no roles, no passwords, and no external IdP. This is intentionally simple because the app is internal.

6. Fuel Helper Logic (canonical)
Definitions:
•	all_in_amount — required, the headline number the customer sees.
•	miles — optional; auto-filled from the lane row if known, otherwise user-entered.
•	fsc_rpm — optional; the fuel surcharge per mile in dollars.
Computed (frontend live preview and backend on save):
hasFsc       = miles  > 0  AND  fsc_rpm  > 0
fsc_total    = miles * fsc_rpm                 (only if hasFsc)
hasLinehaul  = hasFsc AND all_in_amount > 0
linehaul_amount = all_in_amount - fsc_total    (only if hasLinehaul)
Both are rounded to 2 decimals on the server before insert. They are stored on the quote so historical rows do not need to be recomputed.
The "Submit New Quote" panel renders a highlighted summary card showing the math live as the user types:
All-in                          $2,500.00
− FSC (962 mi × $0.78)            −$750.36
─────────────────────────────────────────
Submit linehaul                 $1,749.64    [Copy]
If only miles + fsc_rpm are filled in, the summary card shows the FSC total and an inline hint asking for the all-in.

7. UI Layout — / (Lane Pricing & History page)
The page is a single full-width column with a max width container, top-padded, dark theme matching the Freight DNA brand (jet black bg #0A0A0A, signal gold #FFC333 primary, white text, 4-px radius, Inter font).
7.1 Header
•	Title: "Lane Pricing & History" (with a navigation icon).
•	Subtitle: "Submit new quotes and see what we've previously submitted to customers on this lane."
•	Right side: "Signed in as <name> — Change" pill (opens the welcome dialog to change the name).
7.2 Recent lanes shortcut row
Visible only after at least one search has been performed. A horizontal list of up to 5 pill buttons, each labeled Origin, ST → Dest, ST EQUIP. Clicking re-runs the search. Persisted in localStorage under lm_recent_lanes (deduped by lane key, most recent first).
7.3 Lane search bar (full-width Card)
Form with three fields and a Search button:
•	Origin: city/state autocomplete backed by GET /api/locations.
•	Destination: same.
•	Equipment: select with VAN, REEFER, FLATBED.
•	Search Lane button, disabled until both origin and destination are chosen.
On submit:
1.	The lane is reflected into the URL as ?oc=&os=&dc=&ds=&eq= so the page is shareable.
2.	GET /api/lanes/history is fired.
3.	The lane is pushed onto the recent-lanes list.
On mount, if the URL already carries those params, the search is auto-run.
7.4 Lane summary + share
Under the search bar (only after a search has run): a one-line summary Origin → Destination [EQUIPMENT] with a Copy button on the right that copies the current page URL (with the lane params) to clipboard.
7.5 Upper row — three columns
A. Submit New Quote (Card)
Form (react-hook-form + Zod):
•	**All-in Amount ($) *** — required, number, with a Copy button next to the label that appears once a positive value is entered.
•	**Customer *** — required, free text.
•	Miles — number; auto-filled and made read-only when the lane already has miles cached.
•	Fuel helper (optional) — collapsible, open by default. Contains:
o	FSC RPM ($/mile) — number.
o	The live linehaul-only summary card (see Section 6).
•	Notes (optional) — textarea.
Submit hits POST /api/quotes. On success the form clears, the lane-history query is invalidated, and the customer's fsc_rpm is remembered in localStorage under lm_fsc_rpm:<customer>.
Per-lane draft auto-save: any in-progress values are written to localStorage under lm_draft:<origin_city>|<origin_state>|<dest_city>|<dest_state>|<equipment>. When the user re-searches a lane that has a saved draft, the form is restored and a banner appears: "Restored your in-progress quote for this lane. [Discard draft]". The draft is cleared on successful submit or when the user clicks Discard.
Per-customer FSC RPM memory: typing a customer name that has been quoted before pre-fills the last fsc_rpm they were quoted at. The user can override.
B. Last Submitted Quote to Customer (Card)
Helper text: "What our team previously submitted to the customer — not the live market rate and not the executed load price."
When last_quote is non-null:
•	Big number: $X,XXX all-in.
•	A 12-px trend sparkline of the last quotes (chronological order) with a green ▲ / red ▼ / gray flat indicator and percent change vs the first point.
•	Customer • Date • "Entered by <created_by>".
•	A small mono panel with FSC: $/mile ($total) and Linehaul: $... when those values exist.
•	Notes block (if present).
•	"Older Quotes" — last 5 quotes as $amount Customer Date.
When no quotes exist yet: empty-state with a coin icon and "No quotes recorded for this lane yet — submit the first quote".
C. Live External Market Data (Card)
Source tabs: SONAR (active) and DAT (soon) (disabled, reserved for a later integration). Header strip shows the active source and a "Demo data" badge.
2×2 metric grid (currently demo placeholders, deterministic per lane):
•	Spot RPM — $/mile, "7-day avg".
•	Contract RPM — $/mile, "30-day avg".
•	Load-to-truck — ratio, "Origin market".
•	7-day trend — % change "vs prior week".
Footer note: "Connect a SONAR API key to replace these demo values with live data." The component is shaped so a real fetch can be dropped in without re-doing the layout.
7.6 Middle — Lane Forecast (full-width Card)
Inline SVG chart, ~760×180:
•	Title: "Lane Forecast", description shows equipment + lane.
•	12-week projected $/mile curve (deterministic demo data per lane: base + sine wave + slight upward drift).
•	Horizontal gridlines with $X.XX labels on the left, W1…W12 labels on the bottom (every other tick).
•	Filled area under the line in the primary color at 12% opacity, line stroked at full primary color, dot at every week.
•	"Demo data" badge in the header strip.
•	Footer note inviting integration with SONAR or another forecast feed.
7.7 Bottom — McLeod Load History (full-width Card)
Greyed Card with a truck icon:
•	Title: "McLeod Load History".
•	Helper text: "Actual loads we ran on this lane — pulled from McLeod".
•	Body: "Coming soon" with a one-line description that this will show executed loads with total charges, carrier, and PUP/DEL dates once McLeod is connected.

8. Frontend Behaviors / Local Storage Keys
Key	Purpose
lm_recent_lanes	JSON array of up to 5 recently searched {origin, dest, equipment} objects.
lm_fsc_rpm:<lowercased customer name>	Last FSC RPM the user submitted for that customer.
lm_draft:<oc>|<os>|<dc>|<ds>|<eq>	In-progress Submit-New-Quote form values for that lane.
URL query string params used for shareable lane links:
•	oc (origin_city), os (origin_state),
•	dc (dest_city), ds (dest_state),
•	eq (equipment).
Both reading on mount and writing on each search/recent-click are implemented.
