I want you to add a new feature to my existing Freight-DNA platform called:

Proactive Available Freight Outreach Engine

This should help us market available freight to the best-fit carriers 2–7 days before pickup, using the shipment and carrier data we already have in the platform.

Important context:
- Freight-DNA already has carrier-side history based on past/booked shipments.
- We already have some scoring / weighted metrics in place for carrier fit.
- I do NOT want this to be a blind mass-blast tool.
- I want this to feel like lane-based capacity building, not just one-off load tendering.
- Some customers should be eligible for this workflow, and some should not, because certain customers have unique requirements.
- I want this feature to work inside the existing platform and reuse current entities, workflows, and email capabilities wherever possible.
- I care about explainability, rep trust, auditability, and operational usefulness more than flashy UI.

Your job:
First inspect the existing codebase and determine the current architecture, data model, freight objects, carrier objects, lane logic, email/outreach logic, and scoring logic before making changes.

Do not start coding immediately.
Start by producing:
1. findings
2. implementation plan
3. exact files to change
4. new tables / fields / endpoints needed
5. risks / assumptions
6. validation plan

Then begin implementation in phases with checkpoints after each phase.

Main product goal:
Given available freight that is not yet booked, the system should identify the top 20–30 carriers for that lane/opportunity, rank them using our existing weighted logic plus any missing fit signals, and support outreach 2–7 days in advance.

Core product behavior:
1. Detect available freight opportunities from current unbooked / open shipments.
2. Group freight intelligently into lane-based opportunities where appropriate, instead of forcing everything to stay as isolated one-off loads.
3. Rank carriers based on best fit for that exact load or lane opportunity.
4. Allow us to control which customers are eligible for this feature.
5. Allow a rep to review, edit, approve, and send outreach.
6. Track outreach outcomes and feed useful responses back into lane/carrier intelligence.
7. Preserve the current scoring/weighted metrics we already use unless there is a clear reason to extend them.

Feature requirements:

A) Customer eligibility / policy controls
Build a place in the platform where admins can decide which customers are included in this workflow.

Each customer should support:
- enabled for proactive available freight outreach: yes/no
- outreach mode: exact load only / lane-building / both
- manual approval required before sending: yes/no
- max carriers to contact
- lead-time preference (2–7 days default, configurable if needed)
- special requirements / exclusions
- do not use automation flag
- approved carrier only mode if applicable
- notes for unique customer handling

This should be easy for ops/admin to manage.
Do not assume all customers can use the same rules.

B) Lane opportunity logic
I do not want the system to think only in terms of singular shipments.
Where appropriate, create a lane opportunity layer that can group open freight by:
- origin / destination
- equipment type
- timing window
- customer
- recurring lane patterns if supported by existing logic

If the current architecture already has a lane object or recurring lane logic, reuse it.
If not, propose the lightest clean way to add this without breaking existing shipment workflows.

The system should support two modes:
- Exact load outreach: “Can you cover this shipment?”
- Lane-building outreach: “We have recurring or upcoming freight on this lane over the next several days and want to identify interested capacity.”

C) Carrier ranking
For each available freight opportunity, rank the top 20–30 carriers using current scoring plus additional fit logic if needed.

Prioritize signals like:
- repeat carriers we already use
- carriers with proven history on the lane
- carriers with similar-lane history
- carriers that run that geography even if we have not used them on that exact lane
- existing relationship depth
- tender acceptance / responsiveness if available
- on-time / service performance if available
- claims / issue history if available
- equipment compatibility
- customer-specific restrictions
- recent usage / overuse logic if relevant
- confidence score / low-data flag where appropriate

Segment the ranked carriers into buckets if possible:
- repeat/proven carriers
- strong fit but underused carriers
- new or exploratory carriers

The UI should clearly show WHY a carrier was recommended.
I want explainability, not a black-box rank.

D) Outreach workflow
Build a workflow that lets a rep:
- open an available freight opportunity
- see recommended carriers
- see why they were selected
- include/exclude carriers
- manually reorder or pin carriers
- choose send mode
- send now or schedule
- optionally do phased outreach (example: top 10 first, then next 10 if needed)
- log all outreach activity

Default behavior should be rep review + approval, not full autonomous sending.

If our current email infrastructure supports draft creation, thread tracking, and logging, reuse it.
If there are email foundation gaps, call them out and implement safely instead of building fragile automation on top.

E) Outreach messaging
Support outreach templates/drafts for both:
- exact shipment outreach
- lane-building / future freight outreach

The content should be flexible enough that if a carrier cannot cover Thursday’s load but has trucks next week, we can still capture them as future interest for that lane.

Do not hardcode one message.
Support structured variables and editable templates.

F) Response tracking / intelligence
Create structured response outcomes such as:
- interested now
- interested next few days
- interested next week
- future interest
- declined
- not qualified
- no response
- booked
- do not contact for this lane

These outcomes should update the lane/carrier relationship history in a useful way without silently overwriting core carrier truth.

I want this to build reusable lane intelligence over time.

G) Guardrails
Do not let the system mass-email carriers without controls.
Add practical safeguards such as:
- customer eligibility required
- approval rules by customer
- send limits
- duplicate outreach prevention
- recent-contact suppression window
- respect carrier opt-out / do-not-contact status if that exists
- audit trail of who sent what and why
- confidence flags when recommendations are based on thin history

H) Metrics / reporting
Add reporting so we can see whether this feature is actually working.

At minimum track:
- opportunities generated
- opportunities sent
- response rate
- positive interest rate
- booked from outreach
- time to first response
- time to cover
- repeat usage from recommended carriers
- future-interest responses that later convert
- performance by customer
- performance by rep
- performance by lane

If a dashboard already exists, integrate into that rather than building a disconnected new reporting area.

I) UI / UX
This needs to feel native inside Freight-DNA, not like a bolt-on admin tool.

Recommended UX:
- a queue or list for proactive available freight opportunities
- opportunity detail page or panel
- ranked carrier recommendations with score explanations
- customer eligibility settings in customer admin/settings
- outreach history and response states visible in the opportunity and carrier context
- easy navigation between shipment, lane, customer, carrier, and outreach thread

Keep the UI operationally clean and fast.
Prioritize rep usability over visual complexity.

Implementation instructions:
- Reuse current models, services, and UI patterns wherever reasonable.
- Do not create duplicate carrier, customer, shipment, or lane systems if one already exists.
- Keep architecture cohesive with the rest of Freight-DNA.
- Prefer additive, safe refactors over broad rewrites.
- Preserve backwards compatibility where possible.
- Add migrations carefully.
- Add comments only where necessary.
- Validate data assumptions before coding.
- If current naming is inconsistent, recommend a clean naming approach before implementing.
- If you identify related dead code or duplicated workflows, note them separately but do not go on a cleanup tangent unless it directly blocks this feature.
