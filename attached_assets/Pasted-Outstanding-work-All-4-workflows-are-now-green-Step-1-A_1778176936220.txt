Query 1: Email-derived pollution (last 14 days)
Question: How many AF rows reference companies with is_email_derived = true?

sql
-- Query 1: Email-derived customer pollution in Available Freight (last 14 days)
SELECT 
  COUNT(*) AS af_rows_email_derived,
  COUNT(DISTINCT fo.company_id) AS unique_email_derived_companies,
  COUNT(DISTINCT fo.org_id) AS orgs_affected
FROM freight_opportunities fo
JOIN companies c ON fo.company_id = c.id
WHERE c.is_email_derived = true
  AND fo.created_at >= NOW() - INTERVAL '14 days'
  AND fo.status IN ('open', 'awaiting_approval', 'ready_to_send', 'partial');
Decision point:

<50 rows → Phase 1 labels sufficient

50–500 rows → Phase 1 + Phase 2 filter (hide by default)

>500 rows → Escalate; consider emergency quarantine

Query 2: Snooze/SLA desync
Question: How many open AF rows are hidden by snooze while SLA L1/L2 is actively firing?

sql
-- Query 2: Snoozed rows with active SLA escalation
SELECT 
  COUNT(*) AS snoozed_but_sla_active,
  COUNT(*) FILTER (WHERE sla_notified_l1_at IS NOT NULL) AS sla_l1_firing,
  COUNT(*) FILTER (WHERE sla_notified_l2_at IS NOT NULL) AS sla_l2_firing,
  COUNT(DISTINCT owner_user_id) AS owners_affected
FROM freight_opportunities
WHERE snoozed_until IS NOT NULL
  AND snoozed_until > NOW()
  AND awaiting_approval_since IS NOT NULL
  AND status IN ('awaiting_approval', 'ready_to_send');
Decision point:

>10 rows → Phase 2 must include "Snoozed but SLA-active" admin view

>50 rows → Consider auto-pausing SLA when snoozed_until is in the future

Query 3: Import divergence (OneDrive vs Unified Daily Upload)
Question: For today's date, how many (orgId, lane, pickup) triples appear in freight_opportunities but NOT in freight_daily_upload_fact (and vice versa)?

sql
-- Query 3: Import divergence (OneDrive importer vs Unified Daily Upload)
WITH todays_date AS (
  SELECT CURRENT_DATE AS today
),
af_rows AS (
  SELECT DISTINCT
    fo.org_id,
    fo.origin_city || '-' || fo.destination_city AS lane,
    DATE(fo.pickup_datetime) AS pickup_date
  FROM freight_opportunities fo
  CROSS JOIN todays_date td
  WHERE DATE(fo.pickup_datetime) = td.today
    AND fo.status IN ('open', 'awaiting_approval', 'ready_to_send', 'partial')
),
fact_rows AS (
  SELECT DISTINCT
    fdu.org_id,
    fdu.origin_city || '-' || fdu.destination_city AS lane,
    DATE(fdu.pickup_datetime) AS pickup_date
  FROM freight_daily_upload_fact fdu
  CROSS JOIN todays_date td
  WHERE DATE(fdu.pickup_datetime) = td.today
)
SELECT 
  (SELECT COUNT(*) FROM af_rows WHERE (org_id, lane, pickup_date) NOT IN (SELECT org_id, lane, pickup_date FROM fact_rows)) AS in_af_not_fact,
  (SELECT COUNT(*) FROM fact_rows WHERE (org_id, lane, pickup_date) NOT IN (SELECT org_id, lane, pickup_date FROM af_rows)) AS in_fact_not_af,
  (SELECT COUNT(*) FROM af_rows) AS total_af_today,
  (SELECT COUNT(*) FROM fact_rows) AS total_fact_today;
Decision point:

Divergence >10% → Phase 5 reconciler is mandatory

Divergence <5% → Phase 5 can wait; likely explanation is timing (one writer ran, other hasn't yet)

Query 5: Owner attribution drift
Question: How many rows have conflicting owner attribution (where ownerUserId, delegatedToUserId, createdById, approvedById disagree in non-trivial ways)?

sql
-- Query 5: Owner attribution drift
SELECT 
  COUNT(*) AS total_open_af_rows,
  COUNT(*) FILTER (WHERE owner_user_id IS NULL) AS missing_owner,
  COUNT(*) FILTER (WHERE owner_user_id != created_by_id AND delegated_to_user_id IS NULL) AS owner_creator_mismatch,
  COUNT(*) FILTER (WHERE delegated_to_user_id IS NOT NULL AND delegated_to_user_id != owner_user_id) AS delegation_active,
  COUNT(*) FILTER (WHERE approved_by_id IS NOT NULL AND approved_by_id NOT IN (owner_user_id, delegated_to_user_id, created_by_id)) AS approved_by_unknown
FROM freight_opportunities
WHERE status IN ('open', 'awaiting_approval', 'ready_to_send', 'partial', 'covered')
  AND created_at >= NOW() - INTERVAL '14 days';
Decision point:

missing_owner >0 → Investigate importer's owner-stamping logic

approved_by_unknown >10 → Likely fallout from the backfill-incident workflow failure; cross-reference with Task #1127 diagnostic
