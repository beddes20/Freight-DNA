3. ACCOUNT / CUSTOMER NBA CONSUMERS
--------------------------------------------------

Map customer email signals into NBAs.

Examples:
- pricing_request → NBA to respond/quote/follow-up
- objection_* → NBA to address that specific objection
- service_complaint → NBA to perform service recovery
- stalled_thread → NBA to re-engage
- urgency_signal → NBA with high urgency
- new_opportunity → NBA to qualify/expand

Implementation:

A) Extend NBA creation logic with an “email-driven” path:
- Accepts emailSignal + linked account/load/lane.
- Uses a mapping table from intent → { nbaType, outcomeType, severity }.

B) Dedupe rules:
- Only one active NBA per (accountId, threadId, intentFamily), e.g.:
  - pricing_request family: only one open “respond to pricing” NBA per thread.
- If an NBA already exists, refresh its “lastSeenAt” or similar instead of creating duplicates.

C) Explanation:
- For each email-driven NBA, store:
  - source emailSignalId / emailMessageId / threadId
  - a small explanation string:
    - “Customer requested pricing on lane X→Y in thread from 2026-04-10”
- Expose this in any NBA detail view.

- lane_offer → NBA to follow-up on that lane
- price_pushback → NBA to review pricing

Implementation:

  - capacity_available → nbaType: 'carrier_capacity_followup', outcomeType: 'Execute'

C) Explanation:
