Context:
We now have multiple sources of carrier intelligence:
- carrier_lane history from TMS/loads
- carrier_declared / claimed lanes (if present)
- Market Signal + carrier NBAs (Task #187)
- Email Intelligence Layer v1 (Task #190):
  - email_signals for:
    - lane_offer
    - capacity_available / capacity_unavailable
    - new_lane_preference
    - new_equipment_or_region
    - price_pushback
    - service_issue
    - soft_commitment / hard_commitment
- Carrier-side NBAs from market signals and email signals

Today, this intelligence is scattered across tables, logs, and panels.
We want a **single Carrier Intelligence tab** in Carrier Hub that:
- summarizes what we know about each carrier’s lanes, regions, and capabilities,
- surfaces suggestions derived from email & signals,
- and lets users accept/reject those suggestions.

Goal:
Ship Carrier Hub Enrichment v1:
- backend: carrier intelligence summary + suggestions
- frontend: a new tab/section in Carrier Hub
- no destructive auto-writes; staged, explainable suggestions only

--------------------------------------------------
1. DATA MODEL – SUGGESTED CARRIER INTELLIGENCE
--------------------------------------------------

If a suitable staging table already exists (e.g., for carrier declared lanes/preferences), extend it.
If not, add a new table, e.g. carrier_intel_suggestions.

Fields:

- id
- carrierId
- orgId (if relevant)
- sourceType:
  - email_signal
  - manual_user
  - market_nba
  - system_inferred
- signalId / emailSignalId (nullable, FK → email_signals)
- marketSignalId (nullable)
- suggestionType:
  - lane_preference
  - region_preference
  - equipment_capability
  - capacity_available
  - capacity_unavailable
  - price_sensitivity
  - service_risk
  - general_note
- payload JSONB:
  - e.g., { origin: "Phoenix, AZ", destination: "Kent, WA", equipment: "dry van", frequency: "weekly" }
- confidenceScore (0–1 or 0–100)
- status:
  - pending
  - accepted
  - rejected
  - auto_accepted
- createdAt
- updatedAt
- acceptedAt / rejectedAt (nullable)
- acceptedByUserId / rejectedByUserId (nullable)
- comment (nullable, user notes)

Add schema + migration and storage layer methods to:
- insert suggestions
- update status (accept/reject/auto_accept)
- fetch suggestions for a carrier

--------------------------------------------------
2. BACKEND – POPULATE SUGGESTIONS FROM EMAIL SIGNALS
--------------------------------------------------

Wire carrier-side email signals into suggestions:

For each email_signal with actorType=carrier and relevant intent:

- lane_offer / new_lane_preference:
  - create a lane_preference suggestion
- capacity_available:
  - create a capacity_available suggestion with region/equipment/time window if present
- capacity_unavailable:
  - create a capacity_unavailable suggestion
- new_equipment_or_region:
  - create an equipment_capability or region_preference suggestion
- price_pushback:
  - create a price_sensitivity suggestion
- service_issue:
  - create a service_risk suggestion
- soft_commitment / hard_commitment:
  - create lane_preference or capacity_available suggestions with a stronger confidence or special flag

Rules:
- dedupe suggestions per (carrierId, suggestionType, payload) + threadId.
- do not overwrite; add new suggestions only when meaningfully new.

Implementation:
- Add a service that:
  - scans recent email_signals for a carrier,
  - maps them to suggestions,
  - inserts only non-duplicates.

Integrate with:
- email intelligence scheduler, or
- carrier NBA pipeline, so suggestions stay up to date.

--------------------------------------------------
3. BACKEND – CARRIER INTELLIGENCE SUMMARY
--------------------------------------------------

Create a carrier intelligence summary endpoint/service, e.g.:

GET /internal/carriers/:carrierId/intelligence

Returns:

- basic carrier info (name, MC, etc., as needed)
- historical lane summary:
  - top N origins/destinations
  - count of lanes / loads by region/equipment
- declared/preferred lanes (from existing tables if present)
- active Market NBAs relevant to this carrier:
  - type, region/equipment, severity
- email-derived suggestions:
  - pending / accepted / rejected suggestions with explanation

Structure the response as:

{
  carrier: { ... },
  history: {
    topLanes: [...],
    topRegions: [...],
    equipmentMix: [...]
  },
  declaredPreferences: [...],
  marketSignals: [...],
  suggestions: {
    pending: [...],
    accepted: [...],
    rejected: [...]
  }
}

Make this an internal API for now (use the same internal auth pattern as other internal routes).

--------------------------------------------------
4. FRONTEND – CARRIER INTELLIGENCE TAB
--------------------------------------------------

Add a new tab/section in Carrier Hub, e.g. “Intelligence”.

Requirements:

- Show:
  - top lane/region/equipment history snapshots (read-only)
  - declared preferences (existing data)
  - pending suggestions section
  - history of accepted/rejected suggestions (collapsed/secondary)

Pending suggestions UI:

- For each suggestion:
  - show suggestionType (lane_preference, capacity_available, etc.)
  - show key details (origin/dest/equipment/region/time window)
  - show source:
    - “From email thread on 2026‑04‑10 (view)”
    - “From market signal: Southeast dry van demand_surge”
  - show confidenceScore
  - actions:
    - Accept → writes status=accepted (and future tasks will consume)
    - Reject → writes status=rejected
    - optional note/comment

The UI should clearly distinguish:
- factual history (what the carrier has actually done),
- staged intelligence (what we think they prefer or can do),
- and active signals/NBAs (what we should do next).

Do NOT auto-edit the core carrier profile fields on accept in this task unless there is an existing safe pattern; it’s enough for v1 that accepted suggestions are stored and visible.

So:

- ensure accepted suggestions are:
  - easy to query by carrierId and suggestionType,
  - clearly separated from pending/rejected.

Add simple storage helpers:
- getAcceptedLanePreferencesForCarrier
- getAcceptedRegionPreferencesForCarrier
- getAcceptedEquipmentCapabilitiesForCarrier

No direct ranking changes in this PR unless trivial.

--------------------------------------------------
6. DEDUPE & SAFETY
--------------------------------------------------

Implement dedupe rules:

- For suggestions:
  - same carrierId + suggestionType + normalized payload + same threadId ⇒ one suggestion
  - if a suggestion is already accepted, do not create a duplicate pending suggestion; maybe update confidence or lastSeenAt.

- For UI:
  - avoid overwhelming the user with dozens of near-identical suggestions from one noisy email thread.

Be conservative.
When in doubt, create fewer, higher-quality suggestions.
