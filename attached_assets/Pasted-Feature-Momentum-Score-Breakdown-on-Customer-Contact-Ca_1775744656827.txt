Feature: Momentum Score Breakdown on Customer Contact Card

Goal
When a user clicks the Momentum Score on the customer contact card, they should see a clear breakdown of what’s driving that score (touchpoint health, relationship depth, volume, lanes, RFPs, momentum) without leaving the customer view.

This is a read-only explanation layer. No scoring logic changes are in scope.

Current state
- server/growthScoreCalculator.ts computes a 0–100 Account Growth Score (Momentum Score in UI) with:
  - positive buckets:
    - Touchpoint Health (40)
    - Momentum (12)
    - Relationship Depth (18)
    - Volume Signal (12)
    - Lane Breadth (10)
    - RFP & Opportunity (8)
  - risk penalties:
    - stale touch (3+ and 7+ business days),
    - no meaningful conversation in 90d,
    - no 3rd Base / HR,
    - overdue task.
- The score result shape (GrowthScoreResult / GrowthScoreDriver) already exists.
- Customer contact card will now show:
  - Momentum Score (0–100),
  - band label (Primed to Grow / Growth Ready / Stable / At Risk).

What to implement

1) Data: expose a structured breakdown
Extend the existing customer detail API (or an adjacent score endpoint) to expose a structured breakdown for the account’s current score.

Add to the response for the customer/account:
- momentumScore: number
- momentumBand: 'high_expansion' | 'growth_ready' | 'stable' | 'at_risk'
- momentumBreakdown: {
    touchpointHealth: {
      points: number;   // 0–40
      max: number;      // 40
      drivers: {
        businessDayRecency: { points: number; max: number; businessDaysSinceLastTouch: number };
        frequency10BusinessDays: { points: number; max: number; touchesInLast10BD: number };
        meaningfulConversations30Days: { points: number; max: number; count: number };
      };
    };
    momentum: {
      points: number;   // 0–12
      max: number;      // 12
      drivers: {
        daysTouchedLast10BD: { points: number; max: number; daysWithTouch: number };
        trendVsPrior10BD: { points: number; max: number; currentTouches: number; priorTouches: number; trendLabel: 'up' | 'flat' | 'down' | 'reengaging' };
      };
    };
    relationshipDepth: {
      points: number;   // 0–18
      max: number;      // 18
      drivers: {
        hasHomeRun: boolean;
        hasThirdBase: boolean;
        multiBaseContacts: number;
        totalContacts: number;
      };
    };
    volumeSignal: {
      points: number;   // 0–12
      max: number;      // 12
      drivers: {
        hasFinancialData: boolean;
        ytdLoads: number;
      };
    };
    laneBreadth: {
      points: number;   // 0–10
      max: number;      // 10
      drivers: {
        corridorCount: number;
      };
    };
    rfpOpportunity: {
      points: number;   // 0–8
      max: number;      // 8
      drivers: {
        hasActiveRfp: boolean;
      };
    };
    penalties: {
      totalPenalty: number;  // negative number or 0
      components: {
        stale3BusinessDays?: number;
        stale7BusinessDays?: number;
        noMeaningfulConversation90Days?: number;
        noThirdOrHomeRun?: number;
        overdueTask?: number;
      };
    };
  }

Notes:
- The exact field names can follow your existing GrowthScoreDriver shape, but keep the same idea: bucket totals, sub-driver values, and penalty components.
- Do NOT change the core score computation; only surface the data.

2) Endpoint strategy
Preferred:
- Extend the existing customer/account detail endpoint to include the breakdown alongside the score (single call).
Alternative:
- Add a dedicated endpoint, e.g. GET /api/accounts/:accountId/momentum-score, returning { score, band, breakdown }.
- If you go this route, the Customer tab should fetch this when it mounts the customer card, and cache it for that view.

3) UI: Breakdown access from customer contact card
On the customer contact card where you show:

- “Momentum Score: 82 · Primed to Grow”

Make the score area clickable:
- Clicking opens a right-side drawer or modal titled “Momentum Score Breakdown”.
- Use the same drawer pattern as you use elsewhere (e.g., lane panel, carrier sheet).

Basic layout inside the breakdown:

Header:
- Account name
- Big score number + band badge + last updated date

Section 1: Summary
- Short explanation: “Momentum Score is a 0–100 signal based on touchpoint health, relationship depth, volume, lanes, RFPs, and behavioral momentum.”
- One-line band interpretation, e.g., for each band:
  - Primed to Grow: “High activity, strong relationships, and room to win more lanes.”
  - Growth Ready: “Healthy base with room to increase activity and depth.”
  - Stable: “Not in trouble, but activity and depth need work.”
  - At Risk: “Sparse activity or weak relationships. Needs immediate attention.”

Section 2: Bucket breakdown (visual)
- For each bucket, show:
  - label,
  - points / max,
  - a mini bar or progress strip.

Example:

- Touchpoint Health — 32 / 40
- Momentum — 9 / 12
- Relationship Depth — 14 / 18
- Volume Signal — 8 / 12
- Lane Breadth — 7 / 10
- RFP & Opportunity — 5 / 8
- Penalties — −10

Section 3: Key drivers (plain language)
Under each major bucket, show a few bullets with data baked in, e.g.:

Touchpoint Health:
- Last touch: 1 business day ago (grace period)
- Touched 7 of last 10 business days
- 2 meaningful conversations in last 30 days

Momentum:
- Touched 7 of last 10 business days vs 4 of prior 10 (up trend)
- Re-engaging after a period of no activity (if applicable)

Relationship Depth:
- 1 Home Run contact, 2 Third Base contacts
- 5 total contacts on file

Risk / penalties:
- −4: no touch in 3 business days recently
- −6: no meaningful conversation in 90 days
- etc.

The goal is that a manager can glance at this and say, “We’re losing points because of recency/meaningful convos, not because volume is low.”

4) Interaction behavior
- Clicking outside the drawer or pressing Esc closes it.
- The trigger (score area) should have focus/keyboard support.
- The drawer should not be editable in this phase; it’s read-only.

5) Performance
- Avoid recomputing the score on every click if it’s already available:
  - Either use precomputed data from the customer detail response, or
  - fetch once and store in component state.
- If data is still loading, show a skeleton or spinner within the drawer.

6) Empty / fallback states
- If for some reason the breakdown is unavailable but a score exists:
  - Show a message: “Breakdown not available yet. Please try again later.”
- If neither score nor breakdown exist:
  - The trigger should be disabled or show a tooltip: “Momentum Score not yet calculated.”

7) Tests / QA
Add or update tests to cover:

Backend:
- momentumBreakdown is returned for accounts with scores.
- penalties.totalPenalty equals the sum of its components.
- accounts without history still return a safe shape (e.g., all zeros).

Frontend:
- Clicking the Momentum Score opens the breakdown drawer.
- Values in the drawer match the mocked API response.
- Each band maps to the correct summary text and color.
- Accounts without scores show the disabled/“not calculated” state.
- The drawer is keyboard-accessible and closable via Esc.

Out of scope
- Any changes to the actual scoring logic or weights.
- Any NBA rule changes.
- Per-contact scoring — this remains per-account for now.

Deliverables
- Backend changes (API or new endpoint) with updated types.
- Frontend changes: customer card + new breakdown drawer component.
- Tests for both backend and frontend behavior.
- Short note describing any assumptions or limitations.
