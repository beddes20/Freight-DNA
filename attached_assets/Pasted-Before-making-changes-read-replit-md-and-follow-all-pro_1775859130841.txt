Goal:
Implement **Carrier Ranking Consumer v1** so accepted carrier intelligence influences:
1) carrier ranking in Lane Work Queue
2) carrier suggestion explanations
3) bulk outreach targeting
4) light procurement suppression behavior where appropriate

This task should make the system materially smarter without allowing noisy or unverified email-derived data to overpower hard historical evidence.

--------------------------------------------------
1. RANKING PRINCIPLES
--------------------------------------------------

We need a clear signal hierarchy.

Signal strength order should be:

1. Historical executed lane history (strongest)
2. Historical regional/equipment/customer fit
3. Accepted carrier intelligence (accepted suggestions / accepted declared preferences)
4. Active market NBA boosts / recency/contextual availability
5. Pending suggestions / raw unaccepted email signals (should NOT affect ranking directly in this task)

Important:
- Accepted carrier intelligence should help ranking, but should not outrank strong actual exact-lane history.
- Rejected suggestions should never influence ranking.
- Pending suggestions should not influence ranking.
- If accepted intelligence conflicts with proven historical behavior, preserve the stronger historical signal.

--------------------------------------------------
2. CONSUME ACCEPTED CARRIER INTELLIGENCE
--------------------------------------------------

Extend carrierRankingService.ts to query and use accepted carrier intelligence.

Use helper methods such as:
- getAcceptedLanePreferencesForCarrier(carrierId)
- getAcceptedRegionPreferencesForCarrier(carrierId)
- getAcceptedEquipmentCapabilitiesForCarrier(carrierId)

If these methods do not exist yet, add them cleanly to storage/services.

Types of accepted intelligence to consume:

A) lane_preference
- exact or near-exact corridor match should provide a ranking boost
- stronger if lane + equipment both match

B) region_preference
- if lane origin/destination falls within accepted preferred regions, provide a smaller boost

C) equipment_capability
- if equipment matches lane requirements, provide a supporting boost only (not enough by itself to rank high)

D) capacity_available
- if recent and accepted, provide a temporary boost
- if it includes a time window, respect freshness

E) capacity_unavailable
- if recent and accepted, provide a temporary suppression or penalty
- do not remove carrier completely unless there is already an existing suppression pattern

F) price_sensitivity / service_risk
- do NOT heavily change ranking in v1
- at most expose as explanatory caution flags
- do not use service_risk as a hidden hard penalty yet unless existing logic already supports that safely

--------------------------------------------------
6. TEMPORAL / FRESHNESS RULES
--------------------------------------------------

Accepted intelligence should not last forever without context.

Implement freshness rules for accepted intelligence:
- lane_preference / region_preference / equipment_capability:
  - can remain durable unless explicitly rejected later
- capacity_available / capacity_unavailable:
  - should be treated as temporary signals
  - only influence ranking if recent (configurable freshness window)
- hard_commitment / soft_commitment:
  - if represented via accepted suggestions, only provide boost if recent

If the table has timestamps:
- use acceptedAt / updatedAt / createdAt appropriately
