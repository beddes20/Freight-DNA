2. CARRIER EXPOSURE MATCHING
--------------------------------------------------

Create a service (e.g., carrierMarketExposureService.ts) that determines which carriers are a good fit for a given Market Signal.

It should consider:
- Carrier lane history:
  - lanes they have run in the last N days/weeks
  - especially origin/destination regions related to the signal’s scopeKey/scopeType
- Carrier-declared lane preferences / capacity:
  - from your carrier hub intelligence tables
  - from the email-parsed declared lanes if those exist
- Equipment type:
  - match carrier equipment capabilities to signal.equipmentType

Basic version of exposure rules:

For a demand_surge / regional_heat signal:
- A carrier is “exposed/fit” if:
  - they have run loads originating from the signal’s originRegion in the last 90 days
  OR
  - they have declared lane preferences including that originRegion and equipment
  OR
  - they have run similar corridors (e.g., GA→anywhere with matching equipment) frequently

For a demand_capacity_imbalance signal:
- Focus on carriers who:
  - have coverage in the hot region but low recent volume from you (capacity under-utilized),
  - OR have strong relevant history but are not currently in your top outreach lists for that region.

Implement exposure matching as a separate helper/service so we can tune it later.

Rules should be:
- deterministic and explainable
- configured with constants (e.g., lookback windows, min run count, recency cutoff) in one place

3. CARRIER NBA GENERATION SERVICE
--------------------------------------------------

Implement a service, e.g., carrierMarketNbaService.ts, that:

- Fetches active Market Signals:
  - primarily demand_surge, regional_heat, demand_capacity_imbalance
- For each signal, uses the carrier exposure service to:
  - find candidate carriers
  - compute an urgency score based on:
    - signal severity
    - carrier’s fit strength (history count, preference match)
    - recency

- Upserts carrier-side NBAs:
  - one row per (carrierId, marketSignalId, recommendationType)
  - status default: pending
  - store explanation JSON:
    - signal summary: type, severity, percentChange, supportingEventCount
    - carrier fit summary: runs in region, declared lanes, last-run date, etc.

Dedup rules:
- For a given (carrierId, marketSignalId, recommendationType), do not create duplicates:
  - if there is already a pending or in_progress record, update it as needed (e.g., urgencyScore, explanation) but do not create a new row
- If a previous NBA is completed or dismissed:
  - a new *signal* with a new id can create a new NBA in the future
  - but do not re-open the same signal’s NBA once it’s completed/dismissed

Status interactions with signals:
- When a signal resolves:
  - you may leave completed/dismissed records as history
  - for any still-pending carrier NBAs, you can either:
    - leave them as “stale pending” (simpler for v1)
    - or auto-dismiss them
  - choose the simplest option consistent with the customer NBA behavior
