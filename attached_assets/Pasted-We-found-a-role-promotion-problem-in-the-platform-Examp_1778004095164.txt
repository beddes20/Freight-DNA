We found a role-promotion problem in the platform.

Example: TJ was promoted from LM to AM, but it looks like the platform did not fully update his permissions/visibility to match an Account Manager. He still does not have access to things like the Customers tab that an AM should have.

I want this fixed at the role transition / promotion model level — not just a one-off patch for TJ.

Goal

When a user is promoted or changed from one role to another inside the platform, all permissions, navigation visibility, and route/data access should correctly update to the new role.

In plain English:

changing someone from LM → AM should make them behave like an AM everywhere relevant

old role restrictions should not linger

new role access should not be partially applied

What I need you to do
1. Inspect how role changes currently work

Trace the real code path for:

where a user’s role is stored,

how promotions/role changes are applied,

how nav visibility is derived,

how route/page access is enforced,

how data/API permissions are checked,

and whether there is any cached/derived role state that can become stale after a promotion.

I want to know whether this is caused by:

role not actually being updated in the canonical source,

secondary permission flags not syncing,

stale session/auth claims,

stale cached user profile / org membership data,

or nav and backend permission logic using different sources of truth.

2. Define the canonical role/permission model

I want one clear answer to this:

What should happen system-wide when a user is promoted from LM to AM?

Spell out:

which permissions AMs should have,

which nav items AMs should see,

which routes/pages they should be able to access,

and which LM-only restrictions should disappear after promotion.

If the app currently mixes:

raw role field,

feature flags,

team membership,

explicit grants,

or cached claims

then identify the canonical source of truth and the sync/update path that should happen on promotion.

3. Fix promotions so permissions update completely

I want a real promotion contract:

When role changes:

the canonical role updates,

dependent permissions/visibility update,

stale role-derived state is invalidated/refreshed,

and the user’s effective access matches the new role immediately or on next refresh/re-auth in a predictable way.

This fix should cover:

sidebar/navigation visibility

route/page guards

API/data access checks

customer/account area visibility

and any other AM-relevant product surfaces

4. Validate against the TJ example

Use TJ’s LM → AM promotion as the concrete regression case.

I want you to prove:

what TJ’s current effective role/permissions are,

why he still lacks Customers access,

what the broken transition path was,

and that after the fix, a promoted user like TJ gets the expected AM visibility and access.

If you need to repair TJ’s current user state as part of the fix, do it safely and explain exactly what was repaired.

5. Add guardrails/tests

Add tests that cover role transitions, not just static role checks.

I want tests like:

LM user does not see/access AM-only surfaces

when LM is promoted to AM, nav visibility updates correctly

when LM is promoted to AM, Customers tab becomes visible

when LM is promoted to AM, customer list/detail routes are accessible

stale/cached permission state is invalidated or refreshed correctly

AM does not accidentally inherit unrelated admin-only powers

6. Audit for one-off overrides

Check whether there are any one-off user permission overrides or manual grants that can drift from role-based access.

If so:

identify them,

explain whether they are intended,

and make sure promotions do not leave users in partially transitioned states because of mixed permission models.

Deliverables back

Root cause of the failed LM → AM promotion behavior

The canonical role/permission model you are using

Exact files changed

Before/after proof using TJ (or an equivalent promoted AM test case)

Tests added to prevent future promotion drift

Any required operational note, e.g. whether already-promoted users need a one-time repair or re-sync

Constraints

Do not patch only TJ manually and call it done

Do not give AMs blanket admin rights

Fix the promotion/access model so role changes are reliable system-wide

Desired outcome:
When someone is promoted in the platform, their permissions, visibility, and access update correctly and completely to match the new role.
