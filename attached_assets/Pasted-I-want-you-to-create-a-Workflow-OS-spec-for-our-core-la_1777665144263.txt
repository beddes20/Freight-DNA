I want you to create a Workflow OS spec for our core lane/freight surfaces and make it a first‑class artifact that future work must follow.

Surfaces in scope:

Available Freight (AF)

Lane Work Queue (LWQ)

Available Loads

The goal is to stop per-tab drift and define a single operating model for:

ownership filters,

actionable vs stale logic,

selection + bulk actions,

and outreach behavior.

Please:

1) Ownership model (Owner + AM)

Define a single Owner control that we will use consistently across AF, LWQ, and Available Loads.

Requirements:

It must support:

Me (row owner)

My team

Everyone

Assigned to my accounts (Account Manager lens from companies.assignedTo)

Specific person (with indication of their role if useful)

The UI should be one control with AM‑aware scopes inside it, not two separate filters.

Spell out exactly how this control behaves and how it reads/writes from the underlying ownership envelope.

2) Actionable vs stale model

Define shared rules for actionable, soft overdue, and stale freight across AF, LWQ, and Available Loads.

Requirements:

Default view should show what a rep should work now, not every technically open item.

Propose a concrete rule set, e.g.:

Actionable default: today + future, open, not covered.

Soft overdue: recently overdue but still plausibly live.

Stale: beyond a horizon → hidden by default, accessible via explicit chip/filter.

Call out any per-surface nuance, but the underlying model should be consistent.

3) Selection + bulk actions pattern

Define one shared pattern for selecting and acting on rows in these tables.

Requirements:

Select one / many / all visible.

Clear “X selected” feedback.

Bulk include / exclude.

Works the same in AF, LWQ, Available Loads (with only minor context differences).

Specify how keyboard shortcuts and bulk action bars should work if appropriate.

4) Outreach workspace pattern

Define a reusable bulk outreach workspace pattern that can be used in:

AF,

LWQ,

and any carrier‑selection surface.

Requirements:

Global subject/body template with:

per‑cohort messaging (shortlist vs wider pool),

optional per-carrier overrides.

Select all / deselect all, shortlist‑only, pool‑only.

Search/manage selected carriers at scale.

Preview-as-recipient behavior.

Guardrail/excluded carriers clearly explained.

Same interaction grammar everywhere (learn once, reuse).

5) Containers and layout

Recommend where each of these behaviors should live:

inline controls vs bulk action bar vs side‑sheet vs modal.

My bias: once you’re editing messaging for 20+ carriers, it should be a side‑sheet or workspace, not a tiny modal.

6) Cross‑surface mapping

For AF, LWQ, and Available Loads, spell out:

What should be identical (Owner control behavior, actionable/stale rules, selection patterns, outreach model).

What can be context‑specific (additional filters, KPIs, side panels).

7) Implementation + guardrails

Propose:

a recommended rollout order (which surface to fully align first, then second, etc.),

how we will test and guardrail this OS so future work cannot accidentally diverge from it,

and how existing tasks (#900, #901, #902, outreach modal upgrade) should be updated or superseded to align with this spec.
