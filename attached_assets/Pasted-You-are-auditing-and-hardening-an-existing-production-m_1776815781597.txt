You are auditing and hardening an existing production-minded web platform that has had many new features and additions merged in today.
Your mission is not to build random new features. Your mission is to perform the deepest full-platform QA, systems audit, regression audit, workflow audit, and integration audit we have ever done inside this codebase, then fix the issues you find in a careful, organized way.
I want you to think like:
•	a senior QA engineer,
•	a senior product manager,
•	a senior full-stack architect,
•	a power user trying to break the app,
•	and an operations leader who needs the platform to feel seamless, intuitive, and fully connected.
The biggest concern is this:
we added a lot of powerful functionality, and when that happens, features often become fragmented, partially connected, inconsistent, or buggy. I do NOT want this platform to feel segregated. I want all key features to talk to each other properly, share the right data, behave consistently, and feel like one unified operating system.
Your objectives are:
1.	Audit the entire application before making major assumptions.
2.	Find bugs, broken flows, silent failures, weak error handling, bad state transitions, duplicate logic, and inefficiencies.
3.	Verify that all recent additions merged correctly and did not break older functionality.
4.	Verify that data moves correctly across every connected feature, module, page, workflow, modal, form, automation, dashboard, chatbot, search flow, upload flow, filter, notification, and settings area.
5.	Interact with the platform aggressively: click every button, every checkbox, every dropdown, every tab, every modal trigger, every bulk action, every save path, every cancel path, every back path, every delete/archive/edit path, every search/filter/sort path, every export/import path, every message/send path, and every role-restricted path that exists.
6.	Test every possible input and output path you can reasonably find, including edge cases.
7.	Make the UX more intuitive where the platform feels disconnected, confusing, redundant, or overly compartmentalized.
8.	Ensure features are integrated, not siloed.
Execution rules:
•	Do NOT rush into code changes before understanding the system.
•	Start with a structured platform-wide audit and map the current architecture, feature set, routes, components, shared state, APIs, services, database interactions, background jobs, automations, and user flows.
•	Identify all major product areas and create a testing matrix for them.
•	Create a list of all user-facing and system-facing interactions that must be tested.
•	Trace data flow between modules, especially where one feature should influence another.
•	Look for merge conflicts, duplicated components, stale code paths, dead code, mismatched field names, inconsistent schemas, inconsistent naming, inconsistent permissions, and old logic that may still be controlling newer flows.
•	Look for places where the UI appears to work but the underlying state, persistence, or downstream action is wrong.
•	Look for places where a user can enter data one way but that data is not reflected where it should be elsewhere in the platform.
•	Look for partial implementations that stop at the UI layer and never fully connect to backend logic, storage, analytics, notifications, automations, or related modules.
I want this done in phases.
PHASE 1 — SYSTEM UNDERSTANDING
•	Inspect the entire codebase and summarize:
•	app architecture
•	main modules/features
•	routes/screens/pages
•	shared components
•	backend services and integrations
•	state management pattern
•	database/storage model
•	event-driven logic, automations, notifications, and scheduled actions
•	Identify all areas that changed recently if visible from commit history, changed files, timestamps, TODOs, feature flags, or newly added modules.
•	Build a risk map of the most regression-prone and tightly coupled parts of the system.
PHASE 2 — TEST PLAN
Create a very detailed audit and testing plan that covers:
•	functional testing
•	regression testing
•	integration testing
•	UI/UX consistency testing
•	data integrity testing
•	permission/role testing
•	edge-case testing
•	performance inefficiency review
•	error handling review
•	empty state / loading state / failure state review
•	mobile/responsive review if applicable
For every major feature, define:
•	what should happen
•	where data should come from
•	where data should go
•	what related features should update
•	what could break silently
•	what old functionality may be impacted
PHASE 3 — DEEP INTERACTION TESTING
Then execute the audit by systematically exploring and testing the application.
I want you to:
•	click every clickable thing
•	submit every form
•	toggle every toggle
•	open and close every modal
•	test every tab and nested tab
•	test create, read, update, delete flows
•	test cancel and back-out behavior
•	test search, sort, filter, pagination, selection, bulk actions
•	test import/export or upload/download if present
•	test cross-module data propagation
•	test notification flows
•	test chatbot/assistant/tool workflows if present
•	test any automations and triggers
•	test validation and bad input handling
•	test blank, null, duplicate, oversized, malformed, and unexpected inputs
•	test optimistic UI vs actual saved state
•	test refresh persistence
•	test deep links and route transitions
•	test role-based visibility and access behavior
•	test all major workflows end-to-end, not just isolated components
PHASE 4 — INTEGRATION / CONNECTIVITY AUDIT
This phase is critical.
I want you to specifically identify where the platform feels segregated instead of unified.
Audit questions:
•	Do features share context properly?
•	Does activity in one area update related records elsewhere?
•	Are there duplicate entry points for similar actions with inconsistent behavior?
•	Do dashboards reflect newly created or edited records correctly?
•	Do filters/search/results align with underlying source data?
•	Do assistant/chatbot/recommendation/workflow features have access to the right platform context?
•	Are notifications, tasks, statuses, notes, records, analytics, and communication logs synced correctly?
•	Are there places where the same entity is represented differently across modules?
•	Are there separate workflows that should really be consolidated?
•	Are there features that technically work but feel isolated from the rest of the product?
For each issue found, label it as one or more of:
•	bug
•	regression
•	broken integration
•	data mismatch
•	UX confusion
•	performance inefficiency
•	duplicate logic
•	dead/stale code
•	missing validation
•	missing feedback
•	permission issue
•	architecture concern
PHASE 5 — FIXES
Fix issues in a safe and thoughtful order:
1.	critical bugs and broken flows
2.	data integrity and cross-feature sync issues
3.	regression issues
4.	UX confusion and unintuitive behavior
5.	inefficiencies / cleanup / simplification
6.	refactors only where necessary to improve reliability and cohesion
Important:
•	do not introduce big speculative rewrites unless truly necessary
•	prefer targeted, high-confidence fixes
•	preserve existing behavior where it is correct
•	consolidate duplicated logic when it reduces future breakage
•	improve naming and consistency where it helps maintainability
•	tighten validation, error states, and user feedback
•	make features feel interconnected and intentional
PHASE 6 — VERIFY FIXES
After making fixes:
•	re-test all affected areas
•	re-test related upstream and downstream flows
•	confirm no regressions were introduced
•	confirm persistence works
•	confirm UI state matches backend state
•	confirm related modules reflect updates correctly

NON-NEGOTIABLE STANDARDS
•	Be exhaustive.
•	Do not skip areas because they seem minor.
•	If there is a checkbox, button, toggle, dropdown, filter, action menu, modal, send path, or save path, test it.
•	If data can enter the platform, follow where it should propagate.
•	If two features should relate to each other, verify that they actually do.
•	If the UX is technically functional but feels disconnected or fragmented, call that out and improve it.
•	If there are old and new ways of doing the same thing, identify whether they should be unified.
•	Be suspicious of silent failures, stale UI, partial saves, inconsistent naming, and fake completion states.
•	Think deeply, not just broadly.
Before making broad changes, first present the audit plan and risk map. Then execute. Then apply fixes carefully. Then re-test.
The final goal is a platform that feels smooth, unified, intuitive, and production-ready — not a collection of separate features living beside each other.
