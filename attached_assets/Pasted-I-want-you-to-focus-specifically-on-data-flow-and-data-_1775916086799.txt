I want you to focus specifically on **data flow and data shaping** to make the Value Truck app — especially the Lane Work Queue — much faster and more efficient. This is not just React micro-optimizations; I want better patterns for how we fetch, compute, and sync data.

Goal: Reduce total data moved and work done per interaction so the app feels lighter and scales better as data grows.

---

## 1) Split “list” vs “detail” responsibilities

Right now, it feels like some endpoints may be doing too much — returning large, complex payloads even when the UI only needs a small subset.

I want you to:

1. Audit the main LWQ and procurement-related endpoints:
   - lane list
   - carrier suggestions
   - my procurement / task list
   - any “load everything for this view” type endpoints

2. For each, tell me:
   - what fields the **list view** actually renders
   - what extra fields are being returned but not used in the list
   - where we are reusing the same heavy payload for both list + detail

3. Propose and implement a split where appropriate:
   - **List endpoints** return only lean objects:
     - ids
     - labels/titles
     - key scores/status flags
     - a few small display fields
   - **Detail endpoints**:
     - only called when we open the right-hand panel / detail view
     - return the richer, heavier data needed for that panel

Make sure list endpoints don’t drag around:
- full history logs
- large nested objects
- unnecessary related entities

---

## 2) Background pre-compute / denormalized views

A lot of “slow feel” in data-heavy dashboards comes from doing too much computation on each request (e.g., scoring, aggregating, joining a bunch of tables).

I want you to:

1. Identify any API where the request logic:
   - recomputes lane scores, carrier ranking, or heavy aggregates on every request
   - joins many tables or does complex filtering that’s mostly stable over time

2. For those, propose a **pre-compute strategy**, such as:
   - background job that writes a denormalized table or materialized view with:
     - lane_id
     - current lane score and priority
     - pre-ranked carrier ids / primary metrics
     - any heavy aggregates we need often
   - LWQ then reads directly from this pre-computed table/view instead of recomputing on every click.

3. Implement at least one concrete case:
   - Example: pre-compute lane “score + priority + basic metadata” for LWQ.
   - Keep it updated via:
     - periodic job (e.g., every N minutes) and/or
     - on-change triggers where appropriate.

We should move from “compute every time the user visits LWQ” to “compute once in the background, read fast many times.”

---

## 3) Incremental sync instead of full reloads

I do not want LWQ or procurement views to refetch huge datasets every time something small changes.

I want you to:

1. Find places where:
   - we refetch the entire lane list after an update to a single lane
   - we refetch all tasks after updating one task
   - we invalidate broad caches instead of patching

2. Change those flows to:
   - **cursor-based pagination** for lists (lanes, tasks)
   - **patch-in-place on the client**:
     - after an update, update just that lane or task in the cached list
     - do not refetch the whole list unless absolutely necessary

3. Ensure the APIs support this:
   - list endpoints should support:
     - `limit`, `cursor` or `page`, `sort`
     - stable ordering and cursors
   - update endpoints should return the updated record in a shape that makes client-side patching trivial.

---

## 4) Client-side windowing of active data

Even with pagination, it’s easy to keep too much data resident in JS and cause memory / GC / render overhead.

I want you to:

1. Identify any place we keep large arrays of:
   - lanes
   - carriers
   - history rows
   - tasks

2. Make sure we:
   - only keep a **window** of active items in memory (e.g., current page ± 1)
   - discard off-screen pages from the in-memory dataset once they are not needed
   - rely on API pagination + caching to re-fetch pages if needed

The goal is: the browser should not be holding thousands of fully-hydrated lane objects for no reason.

---

## 5) Search / filter efficiency (consider a proper index/search engine)

If we are currently doing:
- heavy filtering
- fuzzy search
- complex sorting over large in-memory arrays or raw SQL scans

I want you to:

1. Audit how LWQ search/filter and carrier search are implemented:
   - Are we filtering in JS after loading a big dataset?
   - Are we doing unindexed SQL `LIKE` searches over large tables?

2. If performance is an issue here, propose a better option:
   - indexed DB search (proper indexes on the columns we filter/sort on)
   - OR a search engine / DB feature designed for this (e.g., Postgres full-text, Atlas Search, etc.)
   - always with pagination and tight result sets

Don’t implement a whole search engine if we don’t need it, but make sure we’re not doing obviously inefficient scans.

---

## 6) Perceived performance around data

As part of this data-flow work, also improve perceived performance:

- Show **skeleton rows** for lanes and carriers instead of blank areas while data is loading.
- Avoid blocking UI while heavy data loads; make the page usable quickly with partial data if possible.
- Consider **pre-fetching** detail data for the currently highlighted/hovered lane so the right panel feels instant.

---

## 7) Tests and guardrails for data efficiency

Add some guardrails so we don’t drift back into inefficient patterns.

Examples:

- API tests:
  - verify list endpoints return only expected fields (no accidental huge payloads)
  - verify pagination works correctly and limits rows
- Integration tests:
  - selecting a lane does NOT trigger a full LWQ refetch
  - updating a task/lane patches the list instead of refetching everything
- Optional:
  - simple checks on typical response sizes for key endpoints (e.g., lane list JSON stays under X KB for Y lanes in dev data)

---

1. A summary of:
