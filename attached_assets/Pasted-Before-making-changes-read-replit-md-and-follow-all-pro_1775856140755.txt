Goal of this task:
Ship **Lane Work Queue v2.1** focused on:
1) Instrumentation and observability for real-world performance
2) A debug mode to understand ranking decisions per lane
3) A small tuning pass driven by metrics + better guardrails

We are NOT changing the overall UX or introducing major new concepts.
We are making what’s built:
- observable,
- debuggable,
- easier to tune without regressions.

--------------------------------------------------
1. METRICS & LOGGING
--------------------------------------------------

Add lightweight, structured metrics for the Lane Work Queue carrier suggestion engine, focusing on high-frequency lanes.

We want to be able to answer questions like:
- How often is the “obvious” carrier (with strongest exact-lane history) in the top 3?
- How many carriers per HF lane are:
  - exact-lane carriers?
  - strong regional carriers?
  - boosted by market NBA?
- Are we over/under-using HF bulk outreach (e.g., “Select top 30”)?

Implementation:

For each call to rankCarriersForLane (or the carrier-suggestions endpoint), log:

- laneId
- orgId
- isHighFrequencyLane (true/false)
- number of suggestions returned
- count of carriers where:
  - exactLaneHistory.runsOnThisLane > 0
  - regionalHistory.runsInRegion > 0
  - hasMarketNbaBoost === true
- for top 3 carriers:
  - carrierId
  - fitScore
  - exactLaneRunCount
  - hasMarketNbaBoost

Use a structured logging pattern already used elsewhere (e.g., JSON logs).
Do NOT dump large blobs or email content.

Optional: If we already have a metrics/telemetry abstraction, use it; otherwise, start with structured logs we can query later.

--------------------------------------------------
2. DEBUG MODE FOR RANKING
--------------------------------------------------

Add a **debug mode** that can be toggled by an internal flag or query parameter for the carrier-suggestions endpoint.

Behavior when debug mode is enabled (internal only, behind auth or a special header/query):

- Include a `debug` object per lane suggestion response that contains:
  - the raw score components for each carrier (as available from the ranking engine)
  - the exact-lane / regional / customer-history / outreach / market-NBA sub-scores
  - whether HF-specific adjustments were applied
  - any applied floors/boosts

We do NOT need to expose this debug info to normal users.
It can be available via:
- a querystring flag like `?debug=true`, or
- an internal-only header.

Make sure:
- the debug payload is reasonably small and safe (no PII emails),
- and the type layer/API responses are updated accordingly.

--------------------------------------------------
3. TUNING HOOKS & CONFIG
--------------------------------------------------

Refactor ranking for high-frequency lanes so key weights are driven by config instead of literal constants.

Specifically, externalize into a config module (or config object):

- HF exact-lane score floor thresholds:
  - e.g., minExactLaneRunsForFloor
  - hfExactLaneFloorScore
- Market NBA boost weight:
  - e.g., marketNbaBoostPoints
- Any HF-specific penalties/bonuses we’re applying

The goal:
- Make it easier to adjust numbers later with minimal code changes.
- Keep config values documented in one place.

Add a small comment block documenting each config field and its rationale.

--------------------------------------------------
4. SAFETY & GUARDRAILS
--------------------------------------------------

We want HF ranking to be aggressive but not insane:

Add or verify:
- A guard that prevents HF lanes from returning an empty suggestion list when we have known history or bench carriers.
- A maximum per-lane candidate count (already present for top 30 HF; ensure the backend honors the same limit or a configurable limit).
- Reasonable behavior when no exact-lane history exists:
  - HF logic should fall back gracefully to regional/customer signals.

Where feasible, add tests for:
- HF lane with no exact-lane carriers → decent regional suggestions still appear.
- HF lane with one clear exact-lane carrier → that carrier is in the top 1–3.
- HF lane with multiple exact-lane carriers → they occupy the top slots before pure regional-only carriers.

5. TESTS
--------------------------------------------------

Add/extend tests to cover:

1) Metrics/logging:
   - We can’t assert on log output in detail, but:
     - ensure the logging path is executed without throwing.
     - where possible, use a testable logger abstraction to confirm data shape.

2) Debug mode:
   - With debug enabled, response includes the debug object.
   - Without debug, the debug object is omitted.

3) HF tuning behavior:
   - For HF lanes:
     - exact-lane carriers with ≥ minExactLaneRunsForFloor have fitScore >= hfExactLaneFloorScore.
   - For non-HF lanes:
     - ranking remains unchanged except for intended minor differences (if any).

4) Guardrails:
   - HF lane with no history returns non-empty list when region data exists.
   - HF lane with both history and Market NBA carriers reflects both:
     - exact-lane carriers at top,
     - hasMarketNbaBoost pulled through the explanation/debug information.
