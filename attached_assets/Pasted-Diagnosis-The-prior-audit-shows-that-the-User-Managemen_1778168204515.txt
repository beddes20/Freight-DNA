a) Classification buckets (for users)
We should standardize on at least these user buckets:

Junk users

Obvious garbage: foo@test.com, user+test@…, asdf@example.com, placeholders created during early dev or experiments.

No real activity (no notes, assignments, touches, deals, freight, meetings).

Safe long-term state: quarantined or hard-removed after dependency check and archive.

Demo / fixture users

Seeded by demo scripts (seed-demo-org, seed-directors, seed-jbs-*).

May own demo deals, contacts, freight, meetings in demo orgs.

Safe long-term state: clearly labeled demo, isolated by org, never mixed into real org roster or analytics.

Service / shared-inbox accounts

Mailboxes like rfq@, tenders@, inbox+team@, integrations service users.

May legitimately own emails or carry integration tokens, but should not show as reps or count toward headcount.

Safe long-term state: flagged as service accounts, isolated in their own section or filtered from default views.

Real-and-active users

True employees or contractors actively using the system (recent login, active assignments, owned accounts, freight touches, meetings).

These form the “trusted roster.”

Real-but-inactive users

Former employees, decommissioned reps, or users with past but not current activity (no recent login, no current assignments).

Safe long-term state: soft-deactivated (cannot log in, cannot own new work, but history preserved).

Uncertain users

Edge cases where email/domain looks real but patterns resemble demo/test, or where there is light activity that could be demo.

These should go into a review queue rather than being auto-treated as junk or real.

c) Quarantine → review → merge → archive / deactivate workflow
For users, we usually do not merge identities (that is more common with contacts/companies), but we may need to merge roles or transfer ownership. The lifecycle should look like:

Quarantine detection (no writes to users yet)

Offline job or one-off query identifies candidate junk/demo/service users based on email patterns, domains, creation source, seed scripts, and zero-activity checks.

Mark them in a “User Roster Health” report (read-only) so admins can see the impact.

Flagging (non-destructive)

Add/update user flags: is_demo, is_service_account, is_quarantined, is_active=false where appropriate.

For uncertain cases, apply is_quarantined=true and route them to an admin review queue inside User Management.

Admin review UI

In User Management, add a “Review queue” section/toggle that shows only users with is_quarantined=true or is_demo/is_fixture without a known demo org.

For each row, allow admins to:

Mark as Real user → clear quarantine flags, set is_active=true.

Mark as Demo / fixture → set flags; optionally move to a demo org or ensure they’re restricted to demo-only data.

Mark as Service account → set is_service_account=true and remove them from standard rep-facing rosters.

Mark as Junk → move to “ready-to-deactivate” list.

Deactivation / archive (soft-delete, not hard delete)

For junk or ex-employees:

Set is_active=false, deleted_at, deleted_by, delete_reason.

Remove ability to log in and to be assigned new work (ownership, leads, lanes, etc.).

Keep history intact for audit (notes, freight, deals).

Ownership reassignment (if needed)

Provide bulk actions to reassign owned entities (accounts, opportunities, lanes) from a user being deactivated to a new owner before deactivation.

Only after reassignment and soft-deactivation should we consider hard delete—and even then, preferably never for real users.

Merge (rare)

If someone managed to create a duplicate user identity (same person, two accounts), support:

Reassign all downstream references from source → target user.

Soft-delete source user with reason merged_into:<target_id>.

This should be a separate admin-only tool, not part of routine cleanup.

d) Logic: which users appear in the main User Management list
Define clear rules for what shows by default on the User Management page:

Included in main list (default view):

is_active=true

is_quarantined=false

is_demo=false

is_service_account=false

Excluded from main list, but available via filters/tabs:

Demo users → appear under a “Demo/Fixture users” filter/tab.

Service accounts → appear under a “System/Service accounts” filter/tab.

Quarantined users → appear only under “Review queue.”

Archived/deactivated users (soft-deleted) → appear only when “Show archived” is toggled, with a clear “Deactivated” label and disabled actions.

This preserves trust: when a leader opens User Management in normal mode, they see only real, active humans they expect to manage.

Changing user visibility and flags will affect org charts, RBAC, financial attribution, team performance, and any “active user” metrics, so we must treat this as cross-tab work even if the UI entry point is User Management.

Mislabeling demo or service users as real (or vice versa) can either pollute reporting or cut off important system behavior (e.g., shared inbox processing), so the review queue and clear labeling are critical.
